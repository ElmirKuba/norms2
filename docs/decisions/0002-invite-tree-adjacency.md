# ADR-0002: Дерево приглашений — adjacency list + recursive CTE за портом

- **Статус:** accepted
- **Дата:** 2026-05-29
- **Решает:** Elmir + ToT в [`../archive/decision-map.md`](../archive/decision-map.md) (пункт D2)
- **Контекст-теги:** [domain] [db] [architecture]
- **Частично заменён [ADR-0014](./0014-invitation-edge-table.md):** физическое хранение дерева переехало с колонки `accounts.inviter_id` в отдельную таблицу `invitations`. **Порт `InviteTreeRepository` и рекурсивный CTE — остаются в силе.**

## Контекст
Каждый пользователь имеет 0..1 пригласившего (0 — seed-юзеры и регистрации при `FREE_REGISTRATION`). Это лес деревьев, **append-only**: пригласивший фиксируется в момент регистрации и никогда не меняется. Хранилище должно обслуживать: прямых детей (кого пригласил), родителя (кто пригласил), проверку «X — предок Y?» (авторизация бана) и перечисление всего поддерева X.

## Решение
Хранить связь как **adjacency list** — один self-FK столбец `inviter_id` (nullable для корней). Обход дерева (предок/поддерево) — через `WITH RECURSIVE` CTE. Всё дерево спрятано за доменным портом `InviteTreeRepository` с операциями вида `isAncestor(x, y)`, `listSubtree(x)`, `listChildren(x)`, `getParent(x)`. Домен и use-cases работают только с портом и не знают про CTE/SQL.

## Альтернативы
- **Materialized path / `ltree`** — быстрый префикс-поиск поддерева без рекурсии, но требует денормализованного столбца пути, который надо держать консистентным; с UUID путь длинный. Отвергнут: сложность без выигрыша на нашей мелкой глубине.
- **Closure table** — отдельная таблица всех пар предок-потомок, максимально быстрое чтение, но самая дорогая запись (N строк на регистрацию) и избыточна для дерева с одним родителем без перемещений. Отвергнут как преждевременная оптимизация.

## Последствия
- `database.md`: столбец `inviter_id` (self-FK, nullable, индекс), без отдельной таблицы дерева.
- `domain-model.md`: ввести порт `InviteTreeRepository`; авторизация бана выражается как `isAncestor(banner, target)` (см. будущий ADR по D3 — семантика бана в поддереве).
- **Escape-hatch:** если профайлинг покажет, что рекурсивный CTE — узкое место, добавляем closure-таблицу как **новую реализацию того же порта**, без правки доменного слоя. Это и есть требование «заменить инфраструктуру без касания бизнес-логики».
- Глубина дерева в закрытом инвайт-сообществе мелкая → рекурсивный CTE дёшев; материализация поддерева не нужна.
