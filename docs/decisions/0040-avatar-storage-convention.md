# ADR-0040: Хранение аватарок — порт + имя `generateId().<ext>` на загрузку

- **Статус:** accepted
- **Дата:** 2026-06-05
- **Решает:** Elmir (+ Claude Code, при R2.4/R2.5)
- **Контекст-теги:** [backend] [files] [phase1]
- Уточняет: [ADR-0031](./0031-file-storage-uploads.md)/[ADR-0032](./0032-phase1-refinements.md) (диск, `accounts.avatar`; имя `<accountId>.<ext>` → пересмотрено здесь).

## Контекст

ADR-0031/0032 закрепили: аватарка — файл на диске, путь в `accounts.avatar` (одна на аккаунт). Оставались открытыми: схема имени файла, порядок запись↔БД, абстракция над `fs`, валидация типа. Образец из `~/coding/WorksControl2` (обобщённый реестр `files_uploaded`) рассмотрен; для фазы 1 ЛК выбрана простая модель `accounts.avatar` (обобщённый реестр — отложен до НоваСкил, фаза 3, при реальной нужде в N файлах/метаданных).

## Решение

- **Модель:** одна аватарка на аккаунт — путь-строка в `accounts.avatar` (относительный к `content/`, напр. `avatars/<id>.jpg`), как в ADR-0031/0032. Обобщённый files-реестр НЕ вводим сейчас.
- **Абстракция `fs`:** файловый ввод-вывод за портом `AvatarStoragePort` (`save`/`delete`), реализация `AvatarDiskStorage` в `system/storage/`. Домен/use-case `fs` не трогают (задел под S3 — ADR-0031). Это чище образца WC2 (там `fs` прямо в сервисе).
- **Имя файла:** `generateId().<ext>` — **новое на каждую загрузку** (а не `<accountId>.<ext>`): меняет URL при смене аватара → нет залипания кеша браузера; старый файл удаляется после смены БД.
- **Порядок (upload):** новый файл → запись `accounts.avatar` (БД) → удалить старый файл. Сбой записи БД → удалить осиротевший новый файл (компенсация). **Порядок (delete):** БД=null → удалить файл. Принцип: висящий путь на отсутствующий файл хуже осиротевшего файла.
- **Валидация типа — по magic-bytes** (JPEG/PNG/WEBP), НЕ по `mimetype`/имени (подделываемы). Размер — `AVATAR_MAX_BYTES`; интерцептор имеет хард-кап против OOM.
- **Раздача:** `content/` отдаётся статикой под `/content/` (`useStaticAssets`).

## Альтернативы

- **Имя `<accountId>.<ext>` (ADR-0032).** Перезапись по тому же пути → кеш браузера показывает старое; смена расширения оставляет старый файл. Отвергнуто в пользу `generateId().<ext>`.
- **Обобщённый `files_uploaded` (как WC2) сейчас.** Гибко (N файлов/метаданные), но новая сущность/таблица/миграция ради 1:1-аватарки. Отложено до фазы 3 (тот же `AvatarStoragePort`/storage-порт переиспользуем).
- **БД-строка раньше файла (как WC2).** Осиротевшая запись на несуществующий файл при сбое записи. Отвергнуто.

## Последствия

- env: `CONTENT_DIR` (корень загрузок). `accounts.avatar` хранит путь относительно него.
- `deployment.md`: `content/` — в бэкапах, вне git.
- Будущее (фаза 3): обобщённый files-реестр за тем же storage-портом — отдельным ADR.
