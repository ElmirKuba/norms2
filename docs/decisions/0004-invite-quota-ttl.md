# ADR-0004: Приглашения — состояния, квота, TTL, отзыв

- **Статус:** accepted — **механика квоты заменена [ADR-0007](./0007-invite-quota-counter.md)** (состояния инвайта, TTL 3д, отзыв — остаются в силе)
- **Дата:** 2026-05-29
- **Решает:** Elmir + ToT в [`../archive/decision-map.md`](../archive/decision-map.md) (пункт D4 — квота/TTL инвайтов)
- **Контекст-теги:** [domain] [db] [backend]

## Контекст
Инвайт одноразовый, с обязательной причиной ([ADR-0001](./0001-data-minimization-no-pii.md)). Без лимитов создание кодов — вектор абуза (тысячи кодов). Нужны: квота, срок жизни, отзыв — и управляемость без жёсткого хардкода.

## Решение

**Состояния инвайта:** `pending` → `used` (есть `used_by`/`used_at`) | `revoked` (отозван автором) | `expired` (вышел TTL).

**Квота (D4-A):** ограничение на число **одновременно висящих** (`pending`) инвайтов у пользователя.
- Дефолт — `INVITE_MAX_PENDING` из ENV, **значение 3**.
- Персональный override: nullable-поле `users.invite_quota_override`. **Эффективный лимит = `invite_quota_override ?? INVITE_MAX_PENDING`.**
- Использование / отзыв / истечение инвайта освобождает слот (он перестаёт быть `pending`).
- Проверка квоты — серверная агрегация (`count(pending)` у пользователя), идемпотентно.

**TTL (D4-B):** `INVITE_TTL_DAYS` из ENV, **значение 3** (`0` = бессрочно). Инвайт валиден для погашения ⇔ `pending` И `created_at + TTL > now`. Проверка ленивая на чтении; фоновая чистка не обязательна (можно добавить sweep, помечающий `expired`).

**Отзыв:** автор может отозвать свой `pending`-инвайт (`→ revoked`), это освобождает слот квоты.

**Связь с `FREE_REGISTRATION`:** механика инвайтов и квота работают независимо от режима регистрации. В свободном режиме инвайт не обязателен для входа, но остаётся для атрибуции дерева.

## Альтернативы
- **Lifetime-cap** (всего M за всё время) — штрафует давних добросовестных. Отвергнут.
- **Без лимита / без TTL** — проще, но открывает абуз и копит висячие коды. Отвергнут.

## Последствия
- `database.md`: таблица `invites(code, reason, inviter_id, status, used_by, used_at, created_at)`; в `users` — nullable `invite_quota_override`; индекс по `(inviter_id, status)` для подсчёта pending.
- `deployment.md` / ENV: добавить `INVITE_MAX_PENDING` (=3) и `INVITE_TTL_DAYS` (=3).
- **Админ-override отложен:** колонка и логика «эффективный лимит» существуют с фазы 1, но действие «поднять лимит юзеру» требует админ-роли, которой в фазе 1 нет (ТЗ §4.6). Это первый конкретный повод для админ-роли — учтено в пункте **A5** карты решений. До появления роли override не выставляется UI-ом, всем действует дефолт из ENV.
- `domain-model.md`: сущность `Invite` с инвариантами состояний; use-case `CreateInvite` проверяет эффективную квоту; `ConsumeInvite` проверяет валидность (pending + не истёк).
