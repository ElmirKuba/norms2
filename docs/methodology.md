# methodology.md — метод рассуждения Claude Code в этом репозитории

> Как Claude Code думает над задачами Нормисов. Короткая выжимка — в [`../CLAUDE.md`](../CLAUDE.md) (грузится всегда). Здесь — детали и примеры.
>
> Это про **рабочий процесс ассистента**, не про продукт. Те же приёмы как *продуктовая* механика НоваСкила — отдельная история, см. [`../Technical-assignment.md` §2 / Фаза 3](../Technical-assignment.md).

## Зачем формализовать

Проект слоистый и трудно-обратимый: ошибка в домене или контракте тянется через миграции и фронт. Поэтому «подумать вслух → разветвить варианты → перепроверить» — не блажь, а защита от дорогих ошибок. Три приёма ниже закрывают три разные стадии.

---

## 1. Chain-of-Thought (CoT) — пошаговый разбор

**Когда:** перед любой нетривиальной правкой (новая сущность, изменение контракта, миграция, нетривиальный use-case).

**Как:** до кода — явно расписать шаги: что меняется → в каком слое → какие инварианты домена затронуты → что обновить в docs. Это прямое продолжение «Рабочего цикла» из `CLAUDE.md`: не «сразу патчить», а сначала проговорить цепочку.

**Маркер качества:** если не могу выписать цепочку из 3–5 шагов — значит задача недопонята, надо вернуться к `Technical-assignment.md` / спросить Elmir-а.

**Анти-паттерн:** CoT ради мелкой механической правки (переименование, опечатка в доке). Не раздуваем.

---

## 2. Tree-of-Thoughts (ToT) — ветвление на развилках

**Когда:** есть архитектурная развилка с неочевидным выбором (хранение дерева приглашений, стор токенов, reverse-proxy, модель восстановления пароля — см. открытые вопросы C1–C4 в обсуждении и `docs/decisions/`).

**Как:**
1. Сгенерировать **2–4 ветви** решения, а не первую попавшуюся.
2. Каждую оценить по нашим критериям: совместимость со слоистой архитектурой, обратимость, 152-ФЗ, простота, цена сопровождения.
3. Отсечь слабые, явно назвать почему.
4. **Результат ветвления = ADR.** Секции «Решение» и «Альтернативы» в [`decisions/`](./decisions/README.md) — это и есть сериализованное дерево: выбранная ветвь + отсечённые с обоснованием.

Иначе говоря: **каждый ToT над развилкой обязан оставить после себя ADR.** Развилка, решённая «молча в голове», — потеряна.

**Анти-паттерн:** ветвление там, где есть очевидный дефолт или решение уже зафиксировано в ADR. Тогда — просто следуем ADR.

---

## 3. Self-Consistency (SC) — независимая перепроверка

**Когда:** перед тем как пометить пункт в трекере фазы (`impl-phaseN-plan.md`) как `[x]`/✅ для всего, что легко сломать незаметно: контракты DTO, миграции, доменные инварианты, правила доступа (бан, поддерево приглашений).

**Как:** перепроверить результат **другим путём**, а не перечитыванием того же текста:
- контракт — сверить с `api-contracts.md` и с потребителем на фронте;
- миграция — мысленно прогнать up и down, проверить на пустой и непустой БД;
- правило доступа — прогнать через граничные кейсы (сам себя, чужое поддерево, уже забаненный, разбан вышестоящим);
- утверждение «соответствует 152-ФЗ» — сверить с `152fz.md`, а спорное — к Elmir-у (никогда не решаю сам, см. CLAUDE.md).

**Идея:** если два независимых прохода сходятся — уверенность высокая. Расходятся — найдена ошибка, это успех проверки, а не помеха.

**Анти-паттерн:** SC как ритуал на тривиальных правках. Применяем там, где цена тихой ошибки высока.

---

## Как три приёма стыкуются

```
Задача
  │
  ├─ CoT  — разложить на шаги (всегда, кроме тривиального)
  │
  ├─ развилка? ── да ──► ToT — ветви → оценка → ADR в docs/decisions/
  │                нет ─► следуем существующему ADR / дефолту
  │
  └─ перед "done" по рискованному ── SC — независимая перепроверка → todo [x]
```
