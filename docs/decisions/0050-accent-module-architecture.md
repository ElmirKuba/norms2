# ADR-0050: Архитектура раздела «Акцент» — единый модуль + правило слоёв

- Статус: принято (2026-06-15, фаза 2, реш. Elmir)
- Контекст: «Акцент» — большой раздел (~12 областей, ~20 таблиц) и при этом сильно
  переплетён: геймификация слушает события всех областей, дашборд агрегирует всех,
  рекомендатор читает многих. Надо решить раскладку модулей и правило применения
  5-слойки (ADR-0030) так, чтобы не плодить круговые зависимости.

## Развилка (ToT) — раскладка модулей

| Ветвь | Суть | Минус |
|---|---|---|
| A. Под-модули по областям (зонтик импортит) | как домены фазы 1 (account/auth/…) | взаимные импорты модулей → **циклы NestJS** (в ф.1 чинили 3 ADR: 0037/0038/0043), а Акцент переплетён сильнее |
| B. Компромисс (~6 крупных под-модулей) | меньше модулей | те же циклы, просто реже |
| **C. Один `AccentModule` + фича-папки** | все провайдеры в одном модуле, папки — организация файлов | большой массив `providers` (навигация, не корректность) |

## Решение

**Ветвь C — единый `AccentModule` + фича-папки** (`modules/accent/<feature>/` —
`settings/`, `micro-wins/`, `habits/`, `goals/`, …; ORM в `database/`). Все провайдеры
раздела регистрируются в одном `AccentModule` → **межмодульных циклов нет в принципе**
(нет под-модулей, которые импортировали бы друг друга). Это сознательно снимает класс
проблем, который при многомодульной раскладке фазы 1 потребовал `AccessControlModule`
(0037), `BanCoreModule` (0038), `SessionCoreModule` (0043).

**Правило слоёв (применение ADR-0030, «без нюансов»):**
- **Контроллер → use-case — всегда** (use-case = точка приложения/кросс-домена).
- **Кросс-домен — только ВНИЗ:** use-case области A зовёт **domain-service** области B,
  **никогда не use-case B** → граф ацикличен по построению (иначе A.use-case ⇄ B.use-case
  = цикл; ради этого и введён слой domain-services).
- **Свой domain-service / свой порт — по необходимости:** создаём, только если у области
  есть своя логика / свои данные; иначе use-case композирует чужие domain-services, а
  domain-service использует чужие порты. Тонкий use-case над одним domain-service — норма.
- ORM (Drizzle) — только в `database/`; доменные слои зависят лишь от портов (чистые
  границы, как в фазе 1).

## Последствия

- Файлы: `modules/accent/<feature>/{controllers,use-cases,domain-services,adapters,interfaces,dtos}`;
  Drizzle-репо — `database/repositories/accent/...`, схемы — `database/schemas/accent-*.schema.ts`.
- `AccentModule` регистрирует провайдеры всех областей; внутри — следуем правилу «вниз»,
  поэтому циклов провайдеров тоже нет.
- Событийная геймификация (gamification §9) — отдельный механизм развязки (EventEmitter2
  vs свой `DomainEventBus`-порт), решим к подфазе 2.8; на раскладку модулей не влияет.
- Если `AccentModule` со временем станет неподъёмным — можно вынести область в под-модуль
  точечно (осознанно, с учётом цикл-риска), но дефолт — единый модуль.
