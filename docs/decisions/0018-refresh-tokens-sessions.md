# ADR-0018: Refresh-токены и сессии — таблица в Postgres

- **Статус:** accepted
- **Дата:** 2026-05-29
- **Решает:** Elmir (пункт C1)
- **Контекст-теги:** [backend] [db] [security]

## Контекст
Logout = инвалидация refresh-токена (ТЗ §4.2), а чистый JWT отозвать нельзя — нужен серверный стор. Плюс Elmir хочет управление сессиями: кикнуть любое устройство на своём аккаунте (в т.ч. чужой несанкционированный вход) и «выйти везде кроме текущего».

## Решение
**Стор — таблица `sessions` в Postgres** (без Redis):
- `id` (формат [ADR-0016](./0016-primary-key-format.md)), `account_id` FK→accounts, `token_hash` (**хеш** refresh-токена, не плейнтекст), `user_agent` (для опознания устройства в списке сессий), `expires_at`, `revoked_at` (nullable), `created_at`, `updated_at` (≈ last-used при ротации).

**Токены:**
- Access — короткоживущий **stateless JWT** (не хранится; TTL ~15 мин, финал в `backend.md`).
- Refresh — длинный, **в `sessions`**; при обновлении — **ротация** (старый отзывается, новый пишется). Повторное использование отозванного refresh → отзываем все сессии аккаунта (детект кражи).

**Операции:**
- login → создать строку сессии.
- refresh → ротация + обновление last-used.
- logout → `revoked_at` текущей сессии.
- список «мои устройства» → активные сессии аккаунта (с `user_agent`/датами).
- кикнуть конкретную сессию (любую на моём аккаунте — в т.ч. чужой вход) → `revoked_at`.
- «выйти везде кроме текущего» → отозвать все сессии аккаунта, кроме текущей `id`.

## Альтернативы
- **Redis** — быстро, авто-TTL, но лишний сервис в деплое/бэкапах. Против «self-hosted без лишних зависимостей». Отложено до большого масштаба.
- **Без стора** — нет настоящего logout/отзыва, противоречит ТЗ §4.2. Отвергнут.

## Последствия
- **7-я таблица** `sessions`. Итого фаза 1: `accounts`, `secret_qa`, `invite_codes`, `invitations`, `bans`, `security_logs`, `sessions`.
- `database.md`: `sessions` (индекс по `account_id`, по `token_hash`); sweep истёкших.
- `api-contracts.md`: эндпоинты login/refresh/logout + список сессий + revoke (одной / всех-кроме-текущей).
- `backend.md`: TTL access/refresh, ротация, детект повторного использования.
- 152-ФЗ: `sessions` функционально связана с аккаунтом (это не изолированные `security_logs`); `user_agent` — минимально, для опознания устройства самим пользователем.
