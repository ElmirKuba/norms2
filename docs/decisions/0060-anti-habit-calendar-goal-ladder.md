# ADR-0060: «Держусь» — авто-эскалация цели по календарной лестнице

- Статус: принято (2026-07-24, фаза 2, подфаза 2.6.2 · доработки «держусь» D3, реш. Elmir) —
  **обратимо** (фича не задеплоена).
- Контекст: в базе Трека C цель — фиксированное число `target_days`, заданное юзером, и кольцо к нему.
  Лучшие трекеры воздержания (идея из «Bad Habit Break» — берём **только идею**, модель пишем свою,
  CLAUDE.md §4) мотивируют «морковкой»: цель сама «горит» следующим порогом и растёт по достижении.
  Пороги пользователь мыслит **календарно** («месяц», «год»), а не «30/365 дней»: «месяц» = то же
  число в следующем месяце (с клампом к концу месяца, если числа нет). Значит цель — это не «дни», а
  **ДАТА**.

## Решения
1. **Цель = ближайший недостигнутый порог, вычисляемый как ДАТА** (не число дней). Прогресс/кольцо —
   доля от старта серии до даты порога. Вычисляется на чтение (как агрегаты целей, [ADR-0052]) —
   без хранимого счётчика/`version`; долговременный след достижений — события `goal_reached`
   ([ADR-0059]).
2. **Лестница порогов (от даты старта серии):**
   `1д · 3д · 5д · 7д (неделя) · 14д (2 недели) · 21д (3 недели) · +1 месяц · 40д · 50д · +2 месяца ·
   +3 месяца · +полгода (6 мес) · +три квартала (9 мес) · +год` → далее **«морковка»**: предыдущая
   цель + 7 дней, бесконечно.
3. **Два вида смещения порога:**
   - **дневной**: `старт + N дней` (1/3/5/7/14/21/40/50 и «+7д» после года);
   - **календарный**: `старт + N месяцев/лет` с **клампом к концу месяца** («зеркальное число»):
     старт 31 марта → «месяц» = 30 апреля; старт 31 января → 28/29 февраля. Реализация — чистый util
     `addMonthsClamp(date, n)` / `addDays`.
   Движок берёт **самую раннюю пороговую ДАТУ строго после текущего момента** (в TZ аккаунта — границы
   суток важны для «N дней»).
4. **Ручная цель — стартовая ступень, не потолок:** если юзер задал своё число (напр. 7), лестница
   продолжается с ближайшей ступени ≥ него и дальше авто-растёт (2 недели, 3 недели, месяц…). Т.е.
   `target_days` = точка входа на лестницу, а не фиксированный кап.
5. **Достижение порога:** по проходу даты — событие `goal_reached` в таймлайн ([ADR-0059]) + цель
   авто-сдвигается на следующую ступень. В истории видно «цель N достигнута → следующая: M».
6. **После «год»** — «морковка перед осликом»: цель = текущая + 7 дней; достиг → `goal_reached` +
   ещё +7. Бесконечно (никогда не «финиш», всегда есть следующая маленькая цель — anti-burnout,
   [ADR-0049]).
7. **Рассмотренные альтернативы (ToT):**
   - **A (только ручное фикс. `target_days`)** — отвергнута: нет мотивации-морковки, не как лучшие
     трекеры.
   - **B (чисто дневная лестница: 30/60/90/365 в днях)** — отвергнута: пользователь мыслит календарём;
     «месяц» ≠ 30 дней (и это заметно раздражает при старте в конце месяца).
   - **C (календарно-зависимая лестница, цель = следующая ДАТА, вычисление на чтение)** — **выбрано.**

## Последствия
- Смысл `target_days` меняется: seed/стартовая ступень, не жёсткий кап; кольцо считает к **дате**
  следующего порога, а не к фикс. числу.
- Нужен TZ-aware календарный util (`addDays`/`addMonthsClamp`) + генератор следующей ступени —
  чистые функции, покрыть спот-примерами (старт 31-го, високосный февраль).
- Таймлайн получает отметки `goal_reached`; фронт — «морковка» (следующая цель) на кольце/детали.
- **D1 стартер-пак «держусь» авторится ПОСЛЕ** (примеры должны ложиться на эти механики).
- Связано: [[0059-anti-habit-timeline-events]] (`goal_reached`), [[0052-accent-goal-direction-and-computed-progress]]
  (вычисление на чтение), [[0049-accent-product-principles]] (бесконечная маленькая цель), domain-model §7 /
  api-contracts §7 — синхронизировать.
