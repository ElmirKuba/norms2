## P1 — стоит закрыть раньше прочего

1. **`fallbackVersion` («версия на плохой день») введён, но НЕ показан пользователю** — ✔verified
   (помощь людям). Поле есть в данных и в форме создания, но **на карточке/детали цели нигде не
   выводится**. Это прямой повтор паттерна находки 2.4 («RPG-атрибуты собираются, но мертвы»):
   anti-burnout-фича заведена, но **невидима ровно в тот момент, когда нужна** — в тяжёлый день,
   когда человек смотрит на цель и не тянет полную. _Решить:_ показать `fallbackVersion` на детали
   цели (бейдж/строка «на плохой день: …»), возможно — поднимать его в survival-режиме (2.8).
   Быстро, **высокая смысловая ценность** (ядро обещания раздела).

## P2 — важное, но не горит

2. **`uncomplete` привычки НЕ откатывает прогресс цели** — ✔verified (СА + помощь людям).
   `goal_entries` append-only, revoke отложен под 2.9 (как и очки). Сценарий «отметил привычку
   (+20 в цель) → снял отметку → отметил снова» **двоит прогресс цели** (кросс-домен идемпотентен
   только в пределах одного перехода, но uncomplete сбрасывает `transitioned`-гард). Пользователь
   может удивиться «откуда лишние +20». _Решить (2.9, вместе с revoke очков):_ при uncomplete
   удалять/сторнировать связанную запись цели, либо помечать запись источником-задачей для отката.

3. **N+1 на списке/детали целей с rollup** — ✔verified (СА). `list` зовёт `describe()` на каждую
   цель; для rollup-цели `describe()` **рекурсивно** зовёт `describe()` детей (каждый — 1–3
   entry-запроса). Дерево целей → много запросов. На alpha-объёме (целей мало) приемлемо; при росте
   — заменить на батч-агрегат (один SQL с group by по `goal_id`). Отмечено как watch-point ещё на ·9/·12.

4. **Цели изолированы от ежедневного потока** (БА). Раздел самодостаточен, но чтобы увидеть/двинуть
   цель — надо зайти во вкладку. Нет «Сейчас: ближайший шаг к цели» в герое и блока целей в
   кокпите. _Плановое:_ дашборд-агрегатор 2.11 + `Recommender` 2.8 — там цели должны всплыть в
   «делай вот это сейчас», иначе их легко забросить.

## P3 — мелочи / на потом

5. **Нет эндпоинта «подцели цели»** (СА). Фронт `GoalDetailComponent._loadChildren` грузит **все**
   цели (`listGoals()`) и фильтрует по `parentGoalId` клиентски. Целей у юзера немного → терпимо;
   при росте — добавить `GET /accent/goals/:id/children`.

6. **reduce-цели: тон UI не подчёркивает пройденный путь** (помощь людям). Для «снизить» (бросить
   курить: 20→12→0) приятнее видеть «было 20, стало 12 — путь пройден на 40%», чем сухое «12 → 0».
   Данные есть (`startValue`/`currentValue`), нужна доводка копирайта/визуализации под «отказ».

7. **Цель со СВОИМИ записями + добавленные потом подцели** (СА). Если у цели были `goal_entries`,
   а затем завели подцели — прогресс переключается на rollup (среднее % детей), **свои записи
   игнорируются** (не теряются в БД, но «исчезают» из показа). Edge-кейс; при желании — предупреждать
   при добавлении первой подцели или запрещать смешивание.

8. **forecast без дедлайна = null** (БА). Многие цели бессрочны → forecast не показывается (есть
   только `projectedCompletionDate`). Можно мягко подсказывать «поставь срок, чтобы видеть прогноз»,
   но это by-design (прогноз к сроку без срока бессмыслен).

## Итог
Подфаза 2.5 **функционально полна и архитектурно чиста**. Самая ценная быстрая находка — **P1
(показать `fallbackVersion`)**: anti-burnout-обещание раздела сейчас введено, но не видно. P2 (откат
прогресса при uncomplete) и P4 (интеграция в дашборд) — плановые, ложатся на 2.9/2.11. Деплой 2.5.0 —
по решению Elmir (на момент ретро отложен; прод на 2.4.20).
