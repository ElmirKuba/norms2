## Реестр находок

### F#1 — 🔴→✅ Формы «Записать прогресс» и «Добавить веху» не отправляют запрос
- **Симптом (Elmir):** на детали цели вводишь замер/значение или вводишь веху → жмёшь кнопку →
  ничего не происходит, в БД пусто.
- **Доказательство:** в логах nest за сессию — **0 POST** на `/goals/:id/entries` для его целей и
  **0 POST** на `/goals/:id/milestones` вообще (таблица `milestones` пуста во всей БД; `goal_entries`
  для «Тест отжиманий» и «1000 отжиманий» пусты). Браузер слал только GET'ы. Бэк исправен (curl-смоук
  patch 9 писал записи 201).
- **Корень:** в `goal-detail.component.ts` обе формы — `<form (ngSubmit)="…">` **без `[formGroup]`**
  и без `FormsModule` (импортирован только `ReactiveFormsModule`, поля — одиночные `[formControl]`).
  `(ngSubmit)` эмитит **директива формы** (`FormGroupDirective` по `[formGroup]` либо `NgForm` из
  `FormsModule`). Ни одной из них на форме нет → `ngSubmit` **не срабатывает**, `record()`/
  `addMilestone()` не вызываются, POST не уходит. Компилируется без ошибок, в рантайме мёртвая.
  Латентно: `Enter` в одиночном поле мог бы вызвать нативный submit (перезагрузку).
- **Почему не поймали:** «live-verify» всего 2.5 был **curl/API-смоуками**, не кликами в браузере
  (память `tests-deferred`). Прод-сборка лишь проверяет компиляцию. Elmir — первый реальный клик.
- **Охват:** только `goal-detail` (2 формы). Остальное в этом компоненте — правка/удаление записи,
  lifecycle-меню, adopt, удаление вехи — на `(click)`, **работает**. Другие accent-формы
  (логин/регистрация/модалки привычек/микро-побед/целей) используют `[formGroup]` — **не задеты**.
- **Фикс:** обе формы обёрнуты в реальный `FormGroup` + `[formGroup]` + `formControlName`
  (как в логине/модалках). Теперь `ngSubmit` эмитится (клик и Enter), нативной перезагрузки нет.
  `recordForm {value}` и `msForm {title, thresholdValue}`; `editControl` (правка записи) оставлен
  одиночным — он на `(click)`, корректен.
- **Проверка:** prod-build зелёный; рантайм-гарантия — `[formGroup]` всегда даёт `ngSubmit`
  (тот же паттерн, что в работающем логине).
- ✅ **ПОДТВЕРЖДЕНО БРАУЗЕР-СМОУКОМ (Elmir, 2026-06-24):** на «💪 1000 отжиманий» легли **14 записей**
  (current=420) + **4 вехи**; пересчёт `reached` корректен (на скрине при current 11–49 порог 50 был ○;
  при 420 — все ✓). Запись прогресса, вехи и «достигнута» работают end-to-end.

### F#2 — 🟠→✅ После записи прогресса вехи не обновляются без перезагрузки страницы (не SPA-реактивно)
- **Симптом (Elmir, 2026-06-24):** записал прогресс → веха пересекла порог, но `✓`/перечёркивание
  появляется только после **ручного обновления страницы**. Должно обновляться реактивно (SPA).
- **Корень:** `record()` обновляет сигналы `goal` и `entries` из ответа, но **не трогает сигнал
  `milestones`**; `_reloadGoal()` (используется правкой/удалением записи) перечитывает только цель,
  не вехи. А `reached` каждой вехи зависит от текущего значения и вычисляется на момент загрузки
  вех → остаётся устаревшим. (Процент и график обновляются реактивно — бэк возвращает свежую цель
  в ответе на запись; стареют именно вехи.)
- **Охват:** goal-detail, состояние вех после `record()` / `saveEntry()` / `removeEntry()`.
- **Фикс ✅ (2026-06-24):** после мутаций прогресса обновляем вехи реактивно — `record()` на успехе
  зовёт `_loadMilestones()`, и `_reloadGoal()` (правка/удаление записи) тоже. Сигнал `milestones`
  обновляется → OnPush перерисовывает, **без перезагрузки страницы**. Prod-build зелёный; нужен
  браузер-смоук (запиши прогресс через порог вехи → ✓ появляется сразу, без F5).
- **Шире (остаётся в плане ·23.A1):** проаудитить ВСЕ производные вью goal-detail на «устаревание
  после мутации» (% родителя при rollup, дети) — см. принцип SPA-реактивности ([[spa-no-reload-reactivity]]).

### F#3 — 🟠→✅ ИСПРАВЛЕН (превентивно, без репро) Переход между целями может не перезагружать экран
- **Гипотеза:** `goal-detail` берёт `_id = this._route.snapshot.paramMap.get('id')` **один раз в
  инициализаторе поля** (нет подписки на `paramMap`). Angular по умолчанию **переиспy­зует** инстанс
  компонента при навигации `goals/:id → goals/:otherId` (тот же route-config) → конструктор/`_load`
  не перевыполняются → `_id` остаётся прежним → экран покажет **данные предыдущей цели**.
- **Где всплывёт:** клик по **подцели** из родителя (`[routerLink]="['../', c.id]"`), по ссылке на
  родителя/соседнюю цель — без перезагрузки страницы покажет старую цель (вехи/записи/% — чужие).
- **Репро для Elmir:** открой цель с подцелями → кликни подцель (не обновляя страницу) → совпадают ли
  заголовок/проценты с подцелью или остались родительскими? Если родительские — F#3 подтверждён.
- **Рецепт фикса (минимальная churn-версия, выполнить с чистого старта):**
  1. `private readonly _id = this._route.snapshot.paramMap.get('id') ?? '';` → `private _id = '';`
     (мутабельное поле — чтобы все ~15 мест `this._id` остались как есть, без перехода на `_id()`).
  2. В конструкторе вместо `this._load();` — подписка: `this._route.paramMap.pipe(takeUntilDestroyed())
     .subscribe((pm) => { this._id = pm.get('id') ?? ''; <сброс транзиентного состояния>; this._load();
     this._loadMilestones(); this._loadChildren(); })`. (Проверить, не зовёт ли `_load()` уже
     milestones/children — тогда не дублировать.)
  3. **Сброс состояния при смене id:** `entries.set([])`, `hasMore`, `menuOpen.set(false)`,
     `editingId.set(null)`, `recordError`/`msError` `null`, формы `reset()` — иначе утечёт состояние
     прошлой цели.
  4. Импорт `takeUntilDestroyed` из `@angular/core/rxjs-interop` (конструктор = injection-context, ок).
  5. Верификация: prod-build + dev-recompile зелёные; **браузер-смоук за Elmir'ом** (переход
     родитель→подцель и обратно без F5 показывает правильную цель; состояние не течёт).
  Класс тот же, что F#2 — реактивность без перезагрузки ([[spa-no-reload-reactivity]]). Идёт в `·23.F*`.
- **Фикс ✅ (2026-06-24):** `_id` → мутабельное поле; конструктор подписан на `this._route.paramMap`
  (`takeUntilDestroyed`), при смене `id` — `_resetForNewGoal()` (сброс goal/entries/milestones/children/
  меню/правки/форм/ошибок) + `_load()`. Prod-build зелёный. **Браузер-смоук за Elmir'ом:** родитель→подцель
  и обратно без F5 показывают правильную цель; состояние прошлой не течёт.

### F#4 — 🟠→✅ forecast=`behind` при нулевом прогрессе (f=0) вместо `null` (·23.A3, бэк)
- **Корень:** в `goal-progress.util.ts → forecastBlock` блок сравнения темпов выполнялся даже при
  `observedRate=0` (f=0): `0 < requiredRate·(1−ε)` → `behind`. Любая accumulate-цель с дедлайном и
  нулём прогресса сразу показывала «стоит поднажать».
- **Почему баг:** ADR-0052 явно — «forecast = null, если observedRate=0»; и шляпа 3 (тон без вины):
  только что созданную цель нельзя стыдить. (Для reach/reduce без записей f=null — там не всплывало;
  всплывало на accumulate с f=0.)
- **Фикс ✅ (2026-06-24):** весь расчёт forecast/projectedDate обёрнут в `if (observedRate > 0)` →
  при f=0 forecast=null (faithful к ADR). Бэк tsc зелёный. Live-смоук: свежая accumulate-цель с
  дедлайном без записей → forecast=null (не behind).

### F#5 — 🟡 (заметка, НЕ баг сейчас) `pace` для reach/reduce семантически бессмыслен
- `computeGoalProgress` считает `pace = currentValue / activeDays` для всех родов. Для reach/reduce
  `currentValue` — это последний замер (напр. вес 78), и `78/днях` — не «темп прогресса», а мусор.
- **Сейчас не вредит:** на фронте `pace` **не выводится** (проверено grep). Это «спящее» поле.
- **Действие:** если когда-нибудь выводить «темп» — считать его direction-aware (для reach/reduce — Δ
  замера/день), либо не показывать. Занесено, чтобы не выстрелило при добавлении UI темпа.

### F#7/F#8 — 🟠→✅ Молчаливое глотание ошибок в goal-detail (нет фидбэка при сбое)
- **Подтверждено конкретным кейсом (Elmir, 2026-06-24):** «Подцель не создалась в “1000 отжиманий”».
  Причина — НЕ баг бэка: домен корректно вернул **400** с понятным текстом *«У этой цели уже есть свои
  записи прогресса — она не может стать родителем (rollup не смешивается со своими записями)»* (инвариант
  P3#7: цель с 27 записями не может стать rollup-родителем). Но фронт `addSubgoal` имел `error: () =>
  undefined` → **текст выброшен**, пользователь остался без объяснения. Это и есть F#7 во плоти → 🟠.
- **Охват:** в goal-detail на `error: () => undefined` сидели почти все мутации — `addSubgoal`/`openEdit`
  (создание/правка), `_lifecycle` (пауза/архив/снятие/восстановление), `adopt`, `saveEntry`/`removeEntry`,
  `removeMilestone`. `record()`/`addMilestone()` уже показывали ошибку (ок).
- **Фикс ✅ (2026-06-24):** все пользовательские мутации goal-detail теперь сёрфят ошибку бэка через
  `this._modal.error(<контекст>, errorMessage(err))` (готовый `ModalService.error`). Пользователь всегда
  видит причину. Prod-build зелёный. **Браузер-смоук:** «+ Подцель» на цели с записями → модалка с
  понятным текстом (а не тишина). Связь [[spa-no-reload-reactivity]].
- **✅ Проактивный гард (Elmir, 2026-06-24):** «+ Подцель» у цели со своими записями теперь **не открывает
  форму впустую** — сразу показывает модалку «Подцели недоступны» с объяснением (`addSubgoal` проверяет
  `entries().length > 0` ДО открытия формы). Пост-сабмит обработчик оставлен как страховка. Prod-build зелёный.
- **Остаток F#7-класса вне goal-detail:** тихие `error: () => undefined` есть и в списке/дашборде (seed/
  clear/load-доп) — менее критично (там фидбэк не так нужен); занести в общий UX-проход (тост-паттерн).
