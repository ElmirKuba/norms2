# ADR-0010: Флоу регистрации/аутентификации, feature-flags, invite-эндпоинты

- **Статус:** accepted
- **Дата:** 2026-05-29
- **Решает:** Elmir (спек флоу) + пункты F1–F3, C (feature-flags), H (init фронта)
- **Контекст-теги:** [interface] [backend] [frontend] [api]

## Контекст
Нужно зафиксировать, как SPA и бэк ведут себя при регистрации/входе в обоих режимах `FREE_REGISTRATION`, как фронт узнаёт режим и как проверяется код приглашения.

## Решение

### Feature-flags
- Бэк отдаёт **публичный** (до аутентификации) эндпоинт со списком фича-флагов; SPA опрашивает его **при инициализации**.
- Среди флагов — `FREE_REGISTRATION` (bool из ENV). Список расширяемый (closed-shape DTO).

### Регистрация — UX
- **`FREE_REGISTRATION=true`:** кнопка регистрации доступна из публичной части сайта (даже неаутентифицированным). Поля: `alias`, `login`, `password`. Аккаунт создаётся как **корневой** (нет пригласившего) с `registration_source = free`.
- **`FREE_REGISTRATION=false`:** по нажатию «Регистрация» — экран «система для своих, введите код». **Без кнопки отправки:** как только введён код формата `XXXX-XXXX-XX`, фронт сам дёргает проверочный эндпоинт. Ок → фронт держит код в своём стейте/сервисе и прикладывает его к запросу регистрации.

### Регистрация — API
- Payload: `{ alias, login, password, invite_code? }`.
- `FREE_REGISTRATION=true`: бэк **игнорирует** `invite_code` (даже если не пришёл); создаёт корневой аккаунт (`registration_source = free`).
- `FREE_REGISTRATION=false`: `invite_code` обязателен и валиден, иначе ошибка «без кода нельзя». Бэк в **одной транзакции**: гасит код (`invites.used_by = новый юзер`, `status = used`), ставит `users.inviter_id = создатель кода` (правило двойной записи дерева), `registration_source = invite`. Счётчик `invites_remaining` создателя при погашении **не меняется** ([ADR-0007](./0007-invite-quota-counter.md)).
- **Токены при регистрации НЕ выдаются.** Успех → фронт редиректит на страницу входа.

### Вход — API
- Payload `{ login, password }`. Успех → выдаём **access + refresh** (механика стора — пункт C1, ещё открыт) и редиректим в защищённую зону (доступна только аутентифицированным).

### Проверка кода — отдельный эндпоинт
- Только валидация (`pending` + не истёк), **без погашения**. Носит **совещательный** характер для UX.
- Код хранится в БД как простая строка без дефисов (`XXXXXXXXXX`); форматирование `XXXX-XXXX-XX` — только визуальное на фронте.

## Риски и как закрываем
1. **Гонка check→register:** код мог быть погашен/истечь между проверкой и отправкой. Поэтому регистрация **повторно валидирует и гасит код атомарно**; если стал невалиден — внятная ошибка, фронт возвращает на ввод кода. Проверочный эндпоинт ничего не гарантирует.
2. **Энумерация кодов:** проверочный эндпоинт + register + login под **rate-limit** (пункт F5). Коды — длинные, из неоднозначно-несмешиваемого алфавита (рекомендация: без `0/O`, `1/l/I`).

## Доопределено мной (вето приветствуется)
- `users.registration_source enum (free | invite | seed)` — чтобы отличать корни «свободная регистрация» от системных seed-аккаунтов; ты назвал оба случая.
- Алфавит/длина кода — рекомендация выше.

## Последствия
- `api-contracts.md`: эндпоинты `GET /feature-flags`, `POST /auth/register`, `POST /auth/login`, `POST /invites/check`; register не возвращает токены.
- `frontend.md`: на init — запрос флагов; роутинг register по флагу; авто-проверка кода без сабмита; хранение кода в сервисе.
- `database.md`: `users.registration_source`.
- C1 (стор refresh-токенов) — всё ещё открыт, выдача — на входе.
