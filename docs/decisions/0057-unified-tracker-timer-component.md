# ADR-0057: Единый таймер-компонент обратного отсчёта для трекеров

- Статус: принято (2026-07-06, фаза 2, подфаза 2.6 · Трек A / FEAT-H1, реш. Elmir) — **обратимо**
  (чисто фронтовое обобщение; можно вернуть отдельные модалки без изменения контрактов/БД).
- Контекст: фокус-таймер обратного отсчёта (подготовка → отсчёт → готово/отмена, звук off по
  умолчанию) уже реализован для **микро-побед** — `angular/src/app/features/accent/micro-wins/
  micro-win-timer-modal.component.ts` (generic по данным `MicroWinTimerData`: длительность, `prep`,
  подпись). Трек A / **FEAT-H1** (2.6.x) хочет тот же таймер на карточке `timed`-привычки («▶ засечь»,
  длительность = `ladder.currentTarget` с ad-hoc-override; завершение → существующий
  `POST /accent/tasks/:id/complete`, см. `api-contracts`). Вопрос: копировать модалку под привычки
  или обобщить в один компонент.

## Решения
1. **Один общий компонент обратного отсчёта на трекеры.** Вынести таймер-модалку микро-побед в общий
   `accent`-компонент, параметризованный данными (длительность, prep, подпись, коллбэк завершения).
   Потребители — **микро-победы** и **`timed`-привычки** (FEAT-H1); при появлении новых `timed`-подобных
   сценариев подключаются к тому же компоненту. Микро-победный вызов при рефакторинге не должен
   регрессировать (проверить).
2. **«Живой таймер» анти-привычек (2.6 · Трек C) — это ДРУГОЙ компонент, не этот.** Там счёт **ВВЕРХ**
   (`floor((now − startedAt)/сутки)`, `setInterval`, ring к `targetDays`) — иная механика, чем обратный
   отсчёт. Умышленно **не** сводим их в один компонент (см. п.3-B).
3. **Рассмотренные альтернативы (ToT):**
   - **A (копипаст модалки под привычки)** — отвергнута: дублирование логики отсчёта/звука/prep →
     неизбежное расхождение поведения между трекерами.
   - **B (натянуть один компонент и на обратный отсчёт, и на счёт-вверх анти-привычек)** — отвергнута:
     разные механики (down vs up, разовый vs бессрочный, «зачесть задачу» vs «сорвался») → общий
     компонент оброс бы флагами. Держим два простых компонента вместо одного сложного.
   - **C (общий компонент обратного отсчёта для микро-побед + `timed`-привычек)** — **выбрано**.

## Последствия
- Единый источник UX/логики таймера **обратного отсчёта**; поведение (prep, звук, отмена) одинаково
  во всех трекерах, где он применим.
- Изменение — **только фронтовое**: контракты и БД не трогаются (FEAT-H1 зачитывает результат через
  существующий `tasks/:id/complete`, см. [[0056-micro-win-two-axis-taxonomy]]-соседний слой не задет).
- Анти-привычковый «живой таймер» (счёт-вверх) остаётся отдельным компонентом — не путать при
  реализации 2.6 · Трек C.
- Реализация — шаг **H1·2** подфазы 2.6 (`impl-phase2-plan.md`, блок «🧩 2.6 — детальная нарезка»).
