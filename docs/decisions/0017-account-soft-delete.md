# ADR-0017: Жизненный цикл аккаунта — soft-delete (`deleted_at`) + деактивация

- **Статус:** accepted
- **Дата:** 2026-05-29
- **Решает:** Elmir (пункт D8)
- **Контекст-теги:** [domain] [db]

## Контекст
Аккаунт — узел дерева: на него ссылаются `invitations`, `bans`, `invite_codes`. Физическое удаление рвёт дерево и FK. Нужен безопасный способ «удалить/деактивировать».

## Решение
**Только soft-delete, физического удаления аккаунтов нет.** Состояния — двумя nullable-таймстампами на `accounts` (стиль paranoid из Sequelize):

- **`deleted_at`** (null = не удалён). Не-null → аккаунт «удалён»: исключается из обычных выборок (`deleted_at IS NULL`), вход запрещён. **Удалён = удалён:** в админке сайта восстановления НЕТ; технически только администратор БД может занулить `deleted_at`, но это не практика (уточнение Elmir 2026-05-31). Строка остаётся ради истории/целостности дерева. Кто удалился — заводит новый аккаунт по новому приглашению.
- **`deactivated_at`** (null = активен). Самостоятельная **обратимая** пауза по желанию пользователя; реактивация — пользователем (флоу «ваш аккаунт деактивирован, восстановить?» при входе — деталь `ui-ux.md`).

**Вход разрешён ⇔** `deleted_at IS NULL` И `deactivated_at IS NULL` И не забанен (производно из `bans`, [ADR-0012](./0012-bans-derived-status.md)).

«Забанен» и «удалён/деактивирован» — **разные оси**: бан накладывают другие (предки), удаление/деактивацию инициирует сам пользователь.

**Логин остаётся занятым** у soft-deleted/деактивированных: уникальный индекс по `login` — **глобальный** (не partial), чтобы восстановление работало и никто не занял освободившийся логин. Логин не переиспользуется.

**Фаза 1:** само-деактивация и само-удаление — **в скоупе фазы 1** (кнопки в настройках аккаунта).

## Альтернативы
- **Hard-delete** — рвёт дерево/ссылки, требует каскадов. Отвергнут.
- **`status` enum вместо таймстампов** — Elmir выбрал `deleted_at`-стиль; таймстампы дают ещё и «когда».

## Последствия
- `database.md`: `accounts.deleted_at timestamptz null`, `accounts.deactivated_at timestamptz null`; глобальный unique по `login`.
- `backend.md`: дефолтный scope репозитория исключает `deleted_at IS NOT NULL`. UI-восстановления удалённого аккаунта НЕТ (только DBA вручную, не практикуется).
- `domain-model.md`: use-cases `DeactivateMyAccount`, `ReactivateMyAccount`, `DeleteMyAccount`; login-guard учитывает обе оси + бан.
- `ui-ux.md`: настройки — «деактивировать», «удалить»; флоу реактивации при входе.
- 152-ФЗ: ПДн нет, поэтому «право на удаление» не является драйвером; soft-delete достаточен.
