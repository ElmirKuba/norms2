# ui-ux.md — дизайн-язык (сквозной) и экраны фазы 1

> Визуальный язык и экраны ЛК. Дизайн-решения — [ADR-0025](./decisions/0025-ui-ux-design-language.md), тон — [ADR-0022](./decisions/0022-concept-and-philosophy.md). Технически — [`frontend.md`](./frontend.md). Эндпоинты — [`api-contracts.md`](./api-contracts.md).
>
> **Охват:** «Дизайн-язык», токены, тон, доступность ниже — **сквозные** (действуют на все разделы, включая «Акцент»). Раздел «Экраны фазы 1» — только ЛК; экраны разделов — в `sections/<раздел>/ui-ux.md`.

## Дизайн-язык
- **Тема:** тёмная по умолчанию + переключатель на светлую (**CSS-переменные / SCSS**, класс темы на `<html>`; выбор в localStorage).
- **Вайб:** чистый минимал с теплотой — мягкие скругления, тёплый акцент, воздух, живые микротексты. Ламповое/мемное «для своих», но читаемо.
- **Компоненты:** **собственные лёгкие компоненты на чистом SCSS/CSS** — без Tailwind/Bootstrap/UI-китов, полный контроль ([ADR-0025](./decisions/0025-ui-ux-design-language.md)).
- **Модалки — исключение:** диалоги через **`MatDialog` (Angular Material)** — единственная часть Material. Полная механика — [`sections/_shared/modal-system.md`](./sections/_shared/modal-system.md).
- **Графики — второе осознанное исключение:** диаграммы прогресса рисует движок **ApexCharts** (`ng-apexcharts`, MIT, self-host) за **своим** wrapper-компонентом `app-chart` (движок заменяем — остальной код знает только наш контракт `points`/`unit`/`type`), грузится в lazy-чанке. Это сознательное исключение к «без UI-китов» выше — как и `MatDialog` ([ADR-0055](./decisions/0055-progress-charts-apexcharts.md)). Тема графика — из наших CSS-переменных (обе темы). Сами экраны с графиками — в разделе «Акцент» ([sections/accent/ui-ux.md](./sections/accent/ui-ux.md)).
- **Язык:** русский (фаза 1).

## Дизайн-токены (SCSS-переменные / CSS custom properties)
- Палитра: нейтральный фон (тёмный/светлый), 1 тёплый акцент, семантика (success/danger/warning), приглушённые тексты.
- Радиусы: единый набор (sm/md/lg), скруглённые карточки/кнопки/инпуты.
- Типографика: 1 шрифт с весами, шкала размеров; заголовки чуть «характерные».
- Отступы: единая шкала; щедрый воздух.
- Смена темы — переопределением CSS-переменных под классом темы.

## Тон копирайта
- Неформальный, с лёгким юмором, уважительный. Псевдонимы поощряются.
- Пустые состояния и ошибки — с характером, но понятные («тут пока пусто — пригласи своих»).

## Экраны фазы 1

> Поток неаутентифицированного: **cookie-гейт → главная → auth-экраны**. Аутентифицированный попадает в **каркас ЛК** (app-shell) с разделами.

**Вход в приложение**
- **Cookie-гейт** ([ADR-0024](./decisions/0024-cookie-consent-gate.md)): блокирующий, текст про технические куки + ссылка на `/privacy`, кнопки «Я согласен» / «Покинуть сайт» (вторая — redirect).

**Главная (landing)** — для неаутентифицированных, после cookie-гейта
- **Минимум по существу.** Это не маркетинговая воронка, а закрытое «для своих» → без длинных промо-секций: название «Нормисы», одна строка вайба, и **две заметные CTA — «Войти» и «Регистрация»** (контрастные к фону, action-копирайт, крупные ≥44×44). Переключатель темы доступен уже здесь.
- **Единственное явное действие.** Композиция ведёт к одной цели — войти/присоединиться; CTA визуально выделяются над всем (контраст, акцент). Минимальная навигация (ссылки: privacy, может «о проекте»).
- **Invite-режим:** при `freeRegistration=false` CTA «Регистрация» ведёт на экран «введите код» (не сразу на форму).
- **Сознательно НЕ делаем** соц-вход (Google/OTP/биометрия), хотя тренд их советует: это против self-hosted / без облачных зависимостей / без ПДн ([ADR-0001](./decisions/0001-data-minimization-no-pii.md)). Только login+пароль и инвайт.
- **Адаптив:** мобайл — CTA в столбик на всю ширину; десктоп — по центру, в строку.

**Auth** (`features/auth`)
- **register:** поля alias / login / password. При `freeRegistration=false` — сначала экран «система для своих, введите код»: ввод `XXXX-XXXX-XX`, авто-проверка без кнопки, при валидном — пускает к форме ([ADR-0010](./decisions/0010-registration-auth-flow.md)). После успеха — редирект на login (токенов нет).
- **login:** login / password. Ошибка бана — экран «вы забанены»: кто забанил и за что ([ADR-0012](./decisions/0012-bans-derived-status.md)).
- **recover:** ввод login → K случайных вопросов → ответы → новый пароль ([ADR-0008](./decisions/0008-account-recovery-secret-questions.md)).

### UX форм (auth) — из анализа лучших практик 2025/26
- **Пароль:** одно поле + тумблер «показать/скрыть» (доступный с клавиатуры, `aria-pressed`); **без поля «повторите пароль»** — тумблер заменяет подтверждение. Требования к паролю — подсказкой рядом с полем **до** ввода (а не после ошибки).
- **Ошибки — рядом с полем** и **остаются видимыми при исправлении** (не исчезают, чтобы не запоминать); формулировки человеческие, по `error.code` бэка (не «Invalid input»). `aria-invalid` + `aria-describedby` на полях.
- **Минимум полей:** регистрация — только alias/login/password (+ код в invite-режиме); чем меньше трения, тем выше доходимость. Клиентская валидация — зеркало бэка (UX), сервер — источник истины.
- **Сигналы бэка → экраны:** `ACCOUNT_DEACTIVATED` → модалка «реактивировать?» (`POST /auth/reactivate`); `ACCOUNT_BANNED` → экран «вы забанены: кто/за что» (`details.bans`); `QUOTA_EXCEEDED` / `INVITE_INVALID` / `RECOVERY_*` → понятные сообщения.

**Каркас ЛК (app-shell)** — для аутентифицированных
- **Постоянная навигация** по разделам (Профиль / Приглашения / Сессии / Настройки). «Баны» — не пункт меню, а действие в карточке участника. Принципы: «меньше — лучше», **прогрессивное раскрытие** (детали/доп. — под «развернуть»), один экран — одна задача, ключевое читается за ~5 сек (без перегруженных дашбордов).
- **Шапка/угол:** текущий аккаунт (avatar + alias) → меню (Настройки, Выйти) + переключатель темы.
- **Десктоп:** боковая навигация (фикс) или верхняя; **мобайл:** сворачиваемая (бургер / нижняя панель), тач-таргеты ≥44–48px.

**Профиль** (`features/profile`)
- **my-profile (self):** login, alias, дата, кто пригласил, кого пригласил, статус; редактирование alias.
- **user-profile/:login (public):** alias, login, дата. Без действий (фаза 1).

**Приглашения** (`features/invites`)
- **my-invites:** счётчик `invites_remaining`; кнопка «создать код» (поле reason + предупреждение про ПДн); список приглашённых (ник + причина + дата); у активного кода — «отозвать».

**Баны** (`features/bans`)
- Действие «забанить» в карточке приглашённого (поле reason + предупреждение про ПДн); список «мои баны» с «разбанить» (только свои).

**Сессии** (`features/sessions`)
- Список устройств (user-agent, дата, «текущее»); «кикнуть» сессию; «выйти на всех, кроме текущего».

**Настройки** (`features/settings`)
- Секретные вопросы: добавить (из списка или свой + предупреждение про ПДн) / удалить; выбрать K (`recovery_required_count`). Баннер-напоминание «настройте восстановление».
- Деактивировать / удалить аккаунт; флоу реактивации при входе ([ADR-0017](./decisions/0017-account-soft-delete.md)).
- Переключатель темы.

## Доступность и адаптив
- **Брейкпоинты (mobile-first):** проектируем от узкого вверх. Ориентиры: **~480** (телефон), **~768** (планшет / мелкий лэндскейп), **~1024** (лэптоп), **~1280+** (десктоп). SCSS-миксины брейкпоинтов — в дизайн-токенах; добавляем колонки/воздух с ростом ширины, а не наоборот.
- **Тач-таргеты:** интерактив **≥44×44** (Apple) — **48×48** (Google) с зазором между элементами; формы удобны одной рукой на телефоне.
- **A11y (вручную в своих компонентах):** видимый фокус, корректные роли/`aria`, полная клавиатурная навигация, контраст не ниже **AA**, `aria-invalid`/`aria-describedby` на ошибках форм; для модалок — встроенная механика `MatDialog`. A11y — базовое требование, не опция.
- Адаптив закладываем заранее под **Capacitor** (фаза 8) — те же экраны на нативных размерах.

## Предупреждения про ПДн
- На полях `reason` (инвайт/бан) и «свой вопрос» — постоянная подсказка «не указывайте реальные имена, телефоны, адреса» ([ADR-0001](./decisions/0001-data-minimization-no-pii.md)).
