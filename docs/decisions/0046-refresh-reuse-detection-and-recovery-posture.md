- Статус: принято (2026-06-11, разбор TODO фазы 1)
- Контекст: ревизия TODO-маркеров в коде. Три развилки (решены через диалог с Elmir):

  логинов на `/recovery/start`; **#3** чёрный список тем секретных вопросов.
  Дополняет [ADR-0035](./0035-concurrency-control.md) (CAS-ротация),
  [ADR-0041](./0041-session-id-in-access-token.md) (sid в access),

## #4 — reuse-detect реплея ротированного токена

**Проблема.** Ротация CAS перезаписывала `sessions.token_hash`. Предъявление СТАРОГО
(уже ротированного) токена → `findByTokenHash` = null → аккаунт не опознать →
revoke-all невозможен. То есть классический сценарий «токен украли, атакующий
ротировал первым» не детектировался.

### Развилка (ToT)

| Ветвь | Суть | Минус |
|---|---|---|
| A. Ничего | как было | дыра: реплей ротированного токена не ловится |
| B. `family_id` + история строк в `sessions` | копить поколения в той же таблице | sid пришлось бы переносить на family-root; «список устройств» — группировка по family; пёстро |
| **C. Отдельная таблица архива хешей** | `session_token_history` хранит ротированные хеши | +таблица; история растёт (ограничена временем жизни сессии) |

### Решение — ветвь C

Новая таблица **`session_token_history`** (`id`, `session_id`→cascade, `token_hash`
unique, `created_at`). При ротации старый хеш **атомарно (одна tx)** уходит в архив
вместе с CAS-обновлением. При refresh: токена нет среди активных, но есть в архиве →
**реплей ротированного → отзыв ВСЕХ сессий аккаунта** (`revokeAllByAccount`). `sid`
остаётся = `sessions.id` ([ADR-0041](./0041-session-id-in-access-token.md) цел),
«список устройств» не меняется. Миграция `0002` — только `CREATE TABLE`+FK+индексы
(неразрушающая), существующие сессии не рвутся.

Проверено живьём (dev): login → refresh → реплей старого токена → 401 + revoke-all →
свежий токен тоже 401; happy-path цепочки ротаций 200/200, архив по строке на ротацию.

## #2 — анти-энумерация логинов на `/recovery/start`: НЕ делаем

`/recovery/start` отдаёт 409, если логина нет ИЛИ восстановление не настроено — это
раскрывает существование/конфигурацию. **Решение: принять как есть.** Логины —
публичные псевдонимы **by design** (профиль открыт `GET /accounts/:login`, занятость
видна при регистрации), площадка инвайт-онли, на роуте стоит rate-limit (F5.5).
Доп-сигнал «настроено ли восстановление» маргинален, а «единый ответ» ломал бы UX
(восстановление обязано показать реальные вопросы). Полная анти-энумерация
несовместима с «покажи мне мои вопросы» — не платим сложностью за маргинальную выгоду.

## #3 — чёрный список тем секретных вопросов: не на бэке, UI-подсказка

Серверный чёрный список тем (паспорт/ИНН/ФИО…) на **free-text** хрупок (ложные
срабатывания, обходится перефразом), а **ответ хешируется argon2** (в БД не
плейнтекст). **Решение:** без серверного блока — **мягкая UI-подсказка** в форме
добавления вопроса (не использовать ПДн/публично известное, выбирать личное и
неочевидное). Не блокирует ввод.

## Последствия

- Refresh-безопасность доведена до полного reuse-detection (украденный ротированный
  токен → отзыв всех сессий). Цена: +таблица, история растёт в пределах TTL сессии
  (≈число ротаций; чистится cascade при удалении сессии).
- `/recovery/start` и секретные вопросы — осознанно «как есть»; рамка приватности —
  отсутствие ПДн ([ADR-0001](./0001-data-minimization-no-pii.md)), не сокрытие
  публичных по дизайну логинов.
