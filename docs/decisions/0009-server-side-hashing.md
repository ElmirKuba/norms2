# ADR-0009: Хеширование на бэке, фронт шлёт плейнтекст по TLS (фаза 1)

- **Статус:** accepted
- **Дата:** 2026-05-29
- **Решает:** Elmir (пункт E7)
- **Контекст-теги:** [backend] [db] [security]

## Контекст
Обсуждали клиентское хеширование ответов на секретные вопросы. Elmir решил: **на первое время** фронт отдаёт плейнтекст (пароли, ответы) по API, бэк хеширует сам — так проще.

## Решение
- Фронт передаёт **плейнтекст** пароля и ответов на секретные вопросы при регистрации / логине / настройке восстановления / восстановлении. Защита в транзите — **TLS** ([deployment.md], Let's Encrypt).
- Бэк хеширует серверным алгоритмом с **per-record солью**: рекомендуется **argon2id** (иначе bcrypt). Хранится только хеш.
- Сверка: бэк ре-хеширует присланный плейнтекст и сравнивает с хранимым (одинаково для пароля и ответов).
- Клиентского хеширования **нет** (его «хеш-как-секрет»-нюанс не возникает).

## Альтернативы
- **Клиентское хеширование + ре-хеш на бэке** — прячет плейнтекст от бэка, но сложнее (детерминированная соль на клиенте, хеш становится переотправляемым секретом при утечке). Отложено; при необходимости — отдельный ADR (supersede).

## Последствия
- `backend.md`: единый сервис хеширования (argon2id), используется и для пароля, и для `secret_qa.answer_hash`.
- `database.md`: `credentials.password_hash`, `secret_qa.answer_hash` — хеши с солью (формат argon2id самодостаточен).
- `api-contracts.md`: DTO принимают плейнтекст-поля — **только по HTTPS**; никогда не логировать тело этих запросов.
- **«На первое время»** — пометка к пересмотру: при усилении модели угроз вернуться к клиентскому хешированию отдельным ADR.
