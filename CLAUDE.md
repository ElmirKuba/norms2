## Стек и архитектурные принципы

- **Frontend:** Angular 17+, standalone-компоненты, Signals, **чистый SCSS/CSS** (свои компоненты, без Tailwind). SPA. **Angular Material — только `MatDialog`** (модалки), см. [`docs/decisions/0025-ui-ux-design-language.md`](./docs/decisions/0025-ui-ux-design-language.md).
- **Backend:** NestJS, **5-слойная архитектура** ([ADR-0030](./docs/decisions/0030-stack-revision-drizzle-5layer-npm.md), образец — `~/coding/kuba-game`): `controllers → use-cases → domain-services → adapters → repositories` (+ `system`/`interfaces`/`dtos`/`utility-level`). Поток: controller→use-case→domain-service→adapter→repository. **Кросс-доменные вызовы только ВНИЗ:** use-case области A зовёт domain-service ДРУГОЙ области B (не её use-case) → круговой DI исключён. Замена доступа к данным — без правки бизнес-слоёв.
- **БД:** PostgreSQL + **Drizzle** ORM ([ADR-0030](./docs/decisions/0030-stack-revision-drizzle-5layer-npm.md); НЕ TypeORM). Репозитории — слой `repositories`; домен/use-cases про ORM не знают.
- **Идентификаторы (сквозная конвенция, всегда):** PK и все FK — строка формата `uuidv7___unixmillis` (пример `019e7488-0147-7305-9b95-a553f2d00c8e___1780071500548`). Генерация — общий util `generateId()` на бэке и фронте. Подробно — [`docs/decisions/0016-primary-key-format.md`](./docs/decisions/0016-primary-key-format.md). Применяется ко всем таблицам/сущностям во всех фазах.
- **ORM:** Drizzle + drizzle-kit (явные миграции, без auto-push в проде). Слой `repositories` инкапсулирует ORM.
- **Пакетный менеджер:** npm ([ADR-0030](./docs/decisions/0030-stack-revision-drizzle-5layer-npm.md); не pnpm).
- **Docker** для всего (dev + prod), **Let's Encrypt** для TLS, **SSH + Git** для деплоя.
- **Хостинг:** на время разработки РФ допустим, домен `нормисы.рф`; переезд вне РФ — под давлением (см. [`docs/decisions/0023-deployment-jurisdiction.md`](./docs/decisions/0023-deployment-jurisdiction.md)). Щит приватности — отсутствие ПДн, не локация.

### Слоистая архитектура backend (строго)

```
src/
  modules/
    <feature>/
      ... (5 слоёв папками в корне src, см. ниже)
```

**5 слоёв** ([ADR-0030](./docs/decisions/0030-stack-revision-drizzle-5layer-npm.md)) — папками в `src`: `controllers` (контроллеры) → `use-cases` (оркестрация сценария, точка кросс-домена) → `domain-services` (бизнес-логика области) → `adapters` (граница домен↔инфра) → `repositories` (Drizzle). Бизнес-слои не импортируют ORM. **Кросс-домен — только вниз:** use-case области A зовёт domain-service области B (не use-case B) → нет круговой DI. Детали — [`docs/architecture.md`](./docs/architecture.md).

## Рабочий цикл

1. **Уточнить фазу.** Открыть [`todo.md`](./todo.md), найти, к какой фазе относится задача. Если фаза ещё не открыта — сначала закрыть предыдущие.
2. **Прочитать только нужные docs.**
3. **Внести изменения** только в одном слое за раз: либо `nest/`, либо `angular/`, либо `docs/`. Не смешивать в одном коммите backend, frontend и доки.
4. **Обновить документацию,** если изменилось поведение, контракт или структура. Изменение API без обновления `docs/api-contracts.md` — не считается завершённым.
5. **Отметить в `todo.md`,** что пункт выполнен. Если по ходу появились новые подзадачи — дописать туда же.

## Дока отражает ФАКТ, а не только план (правило свежести)

Вся дока — `docs/`, ADR, `todo.md`, `impl-*-plan.md`, **память** — должна описывать **то, чем проект является СЕЙЧАС по факту**, а не только «что будет».

- Пока чего-то **нет** — пишем в будущем времени («будет», «планируется», статус 📋/⬜).
- Как только это **появилось в коде** — в тот же заход переписываем на **настоящее** время («является», «реализовано», статус ✅): описываем не «что надо сделать / как будет выглядеть», а **что оно есть по факту**.
- Это касается и формулировок (будущее→настоящее), и статусов, и memory (`MEMORY.md` + файлы), и `todo.md`.
- При каждом завершённом этапе кодинга: синхронизировать связанные доки/ADR/todo/план/память с реальностью. Расхождение доки с кодом — это баг доки.

Подробности и примеры — [`docs/methodology.md`](./docs/methodology.md). Кратко, всегда:

1. **Chain-of-Thought** — перед нетривиальной правкой расписать шаги: что меняется → в каком слое → какие инварианты домена → что обновить в docs. Не можешь выписать цепочку — задача недопонята.
2. **Tree-of-Thoughts** — на архитектурной развилке генерировать 2–4 ветви, оценивать (слои / обратимость / 152-ФЗ / простота), отсекать. **Итог развилки обязан стать ADR** в [`docs/decisions/`](./docs/decisions/README.md) — иначе решение потеряно.
3. **Self-Consistency** — перед пометкой пункта `todo.md` как `[x]` для рискованного (контракты, миграции, правила доступа, 152-ФЗ) — перепроверить **другим путём**, не перечитыванием. Сошлось — уверенность; разошлось — найден баг.

Тривиальные правки (опечатки, переименования) — без этого ритуала.

## Где НЕ должно быть кода

- ❌ Бизнес-логика в `app.module.ts` или `main.ts` — только bootstrap.
- ❌ Переменные окружения в коде — только через `ConfigService`, описаны в [`docs/deployment.md`](./docs/deployment.md).
- ❌ Импорт ORM (`drizzle`) в use-case/доменных слоях — только через adapters/repositories.
- ❌ Drizzle auto-push в проде — только явные миграции (drizzle-kit).
- ❌ Стейт-менеджеры на фронте (NgRx и пр.) — только Signals + поля класса + rxjs/сервисы при необходимости.
- ❌ `any` в TypeScript. Если тип неизвестен — `unknown` и сужение.
- ❌ ПДн (ФИО, паспорт, реальный email без явной согласия+цели) в БД — см. [`docs/152fz.md`](./docs/152fz.md). Юзеры идентифицируются псевдонимом и логином.

## Когда обращаться к человеку

Спрашивать, а не угадывать, если:

1. Новая сущность не описана в [`docs/domain-model.md`](./docs/domain-model.md).
2. Изменение, ломающее API.
3. Новая внешняя зависимость — сначала проверить, нет ли уже похожей в стеке.
4. Появился новый внешний источник для «вдохновения» — брать **только идеи**, в доку писать с нуля под наш scope; не копировать схемы/DTO/миграции/формулировки напрямую.
5. Вопрос про юрисдикцию / ПДн / соответствие 152-ФЗ — никогда не решать самому.
