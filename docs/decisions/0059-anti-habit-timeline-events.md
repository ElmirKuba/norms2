# ADR-0059: «Держусь» — единый таймлайн событий вместо relapse-only

- Статус: принято (2026-07-24, фаза 2, подфаза 2.6.2 · доработки «держусь» D2, реш. Elmir) —
  **обратимо** (фича «держусь» ещё НЕ задеплоена → прод-данных нет → схему пересобираем свободно).
- Контекст: базовый Трек C (2.6.0) завёл relapse-only таблицу `anti_habit_relapses`
  (`relapse_at`/`attempt_duration_ms`/`trigger_tag`/`note`, ADR — в domain-model §7). Новые механики
  требуют, чтобы «История» была **смешанной хронологической лентой**, а не только срывами:
  - **D2** (перенос/план на будущее): состояние `planned` + событие «перенёс старт с ДАТЫ на ДАТУ,
    продержался N дней» / «планирую начать с ДАТЫ»;
  - **D3** (авто-эскалация цели, [ADR-0060]): по достижении порога — отметка «цель N достигнута».
  Т.е. история = feed из событий разных типов. «Держусь» не в проде → можно переделать без миграции
  данных (лучшее окно для рефактора модели).

## Решения
1. **Заменяем `anti_habit_relapses` → единый `anti_habit_events`** (таймлайн; base Трека C
   переделывается ДО первого деплоя — перегенерить `0026` или добавить `0027`, прод relapse-only
   схему не увидит).
2. **Тип-дискриминатор `type`:** `relapse | reschedule | plan | goal_reached` (varchar-ключ, как
   прочие enum'ы раздела; расширяемо).
3. **Явные типизированные nullable-колонки (НЕ jsonb-свалка — «без скрытой магии»):**
   - общие: `id`, `anti_habit_id` (FK `anti_habits` ON DELETE CASCADE), `type`, `occurred_at`
     (unix ms), `created_at`;
   - `relapse`: `attempt_duration_ms`, `ended_attempt_number`, `trigger_tag?`, `note?`;
   - `reschedule` / `plan`: `from_started_at?`, `to_started_at?`, `held_days?` (сколько продержался
     до переноса; для `plan` при создании — null/старт в будущем);
   - `goal_reached`: `threshold_label` (напр. `неделя`/`месяц`/`год`/`+7д`), `threshold_days`
     (номинал в днях на момент достижения — для сортировки/отображения).
   Незадействованные для типа колонки = null. Индекс `(anti_habit_id, occurred_at)` — keyset-история.
4. **Состояние `planned` — вычисляемое, без новой колонки:** `current_attempt_started_at` может быть
   в БУДУЩЕМ → `now < startedAt` = `planned` (серия ещё не идёт, счётчик показывает «старт через X»,
   не тикает вниз). Настоящий старт = когда время наступит (серия считается от `startedAt` как и
   сейчас). Перенос идущей серии: под CAS (как срыв) завершаем текущую попытку (`reschedule` с
   `held_days`), ставим `current_attempt_started_at` в будущее.
5. **Эмит-события (порт `AccentAntiHabitEventsPort`) остаются про геймификацию (2.9):** `relapsed`
   как есть; при желании 2.9 — `rescheduled`/`goalReached`. Таблица `anti_habit_events` — это
   **история для UI**, порт — **хуки для очков**; концерны разные, но выравнены по типам.
6. **Рассмотренные альтернативы (ToT):**
   - **A (relapse-only + отдельные таблицы на каждый тип)** — отвергнута: 4 таблицы, разрозненная
     лента, сложный merge/сортировка для единой истории.
   - **B (одна таблица + `payload jsonb`)** — отвергнута: прячет форму данных, против «без магии/ORM,
     скрывающей SQL»; типы событий известны и стабильны → явные колонки честнее.
   - **C (единая `anti_habit_events` + `type` + типизированные nullable-колонки)** — **выбрано:** одна
     хронологическая лента, явные поля, чистый рефактор в окне «до деплоя».

## Последствия
- Эндпоинт истории отдаёт смешанные события (`type` + поля); фронт рендерит по типу (иконка/текст:
  срыв / перенос / план / цель достигнута). Cursor-пагинация по `(occurred_at, id)` — как было.
- Base Трека C (домен/репозитории/DTO/фронт истории) **переписывается** под события до деплоя;
  relapse остаётся частным случаем (`type='relapse'`), CAS-логика срыва (ADR-0035) не меняется.
- `planned`-состояние вводит ветку в live-счётчике (счёт-вниз до старта vs счёт-вверх серии).
- Связано: [[0060-anti-habit-calendar-goal-ladder]] (эмитит `goal_reached`), [[0035-concurrency-control]]
  (срыв/перенос под CAS), [[0049-accent-product-principles]] (non-punitive лента), domain-model §7 /
  api-contracts §7 — синхронизировать под события.
