# ADR-0012: Хранение бана — отдельная таблица, статус производный

- **Статус:** accepted
- **Дата:** 2026-05-29
- **Решает:** Elmir (пункт E, уточнение к [ADR-0003](./0003-ban-semantics.md))
- **Контекст-теги:** [domain] [db]

## Контекст
Обсуждали, хранить ли «забанен» флагом в `accounts` или таблицей. Модель [ADR-0003](./0003-ban-semantics.md) (несколько независимых банов, «снять только свой», причина у каждого) не помещается в один флаг.

## Решение
- **Отдельная таблица `bans`:** `id`, `banner_id` (FK→accounts), `target_id` (FK→accounts), `reason`, `active`, `created_at`, `updated_at`. Уникальность активной записи на пару `(banner_id, target_id)`.
- **Никакого флага статуса в `accounts`.** «Забанен» — производное: при логине проверяем `EXISTS(bans где target_id = аккаунт И active)`. Один источник истины, нулевой риск рассинхрона.
- **Сообщение при логине:** забаненному показываем активные баны — **кто забанил и за что** («забанил X за: …»). Банов может быть несколько — показываем все активные.

## Альтернативы
- **Кеш-флаг `accounts.status`** — быстрый login по одной колонке, но дублирование и риск дрейфа; синхронизация при мульти-бане нетривиальна. Отвергнут.

## Последствия
- `database.md`: таблица `bans`, индекс по `(target_id, active)` для быстрого login-чека.
- `domain-model.md`: эффективный бан = `isBanned(account) := EXISTS active ban`; unban деактивирует **свою** запись ([ADR-0003](./0003-ban-semantics.md)).
- `api-contracts.md`: ответ при заблокированном логине отдаёт список активных банов (банивший + причина).
- `ui-ux.md`: экран «вы забанены», показывает банившего и причину.
- 152-ФЗ: `reason` — свободный текст с предупреждением про ПДн ([ADR-0001](./0001-data-minimization-no-pii.md)).
