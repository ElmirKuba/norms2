# ADR-0007: Квота инвайтов — явный счётчик с лимитом за всё время

- **Статус:** accepted
- **Дата:** 2026-05-29
- **Решает:** Elmir + ToT (decision-map, пункт D4 — карта в git-истории)
- **Контекст-теги:** [domain] [db]
- **Заменяет:** механику квоты из [ADR-0004](./0004-invite-quota-ttl.md) (состояния, TTL 3д, отзыв из ADR-0004 — остаются в силе).

## Контекст
В [ADR-0004](./0004-invite-quota-ttl.md) квота считалась как число `pending`-инвайтов (3 одновременно, безлимит за всё время). Elmir уточнил иную механику: явный счётчик, где использованный код **не возвращает** слот.

## Решение
В `users` — колонка **`invites_remaining` (int, default из ENV `INVITE_DEFAULT_QUOTA = 3`)**:

- **Создание кода** → `invites_remaining -= 1`.
- **Отзыв неиспользованного кода** → `invites_remaining += 1` (слот возвращается).
- **Использование кода** (по нему зарегистрировались) → счётчик **НЕ меняется** (слот потрачен навсегда).
- При `invites_remaining = 0` создать новый код нельзя.

Итог: пользователь может привести **не более 3 успешно зарегистрировавшихся** за всё время (плюс оборот за счёт отзыва неиспользованных). Это сознательный жёсткий контроль роста закрытого сообщества.

**Изменения счётчика — в одной транзакции** с созданием/отзывом кода (иначе дрейф). Отозвать можно только `pending`; использованный — нельзя.

**Админ-override:** админ поднимает лимит, просто **увеличивая `invites_remaining`** конкретному юзеру. Отдельное поле `invite_quota_override` из ADR-0004 **не нужно** — счётчик и есть точка управления. Сам админ-экран отложен (нет роли в фазе 1, см. A5).

## Альтернативы
- **Count pending (ADR-0004)** — рост не ограничен, использованные коды освобождают слот. Отвергнут: Elmir хочет жёсткий лимит за всё время.
- **Derived-счётчик** (вычислять, не хранить) — невозможно при семантике «использование не возвращает», т.к. нужен учёт израсходованного за всё время. Поэтому колонка оправдана.

## Последствия
- `database.md`: `users.invites_remaining int not null default 3`; убрать `invite_quota_override`.
- `deployment.md`/ENV: `INVITE_DEFAULT_QUOTA = 3` (переименование `INVITE_MAX_PENDING` из ADR-0004). `INVITE_TTL_DAYS = 3` остаётся.
- `domain-model.md`: use-case `CreateInvite` проверяет `invites_remaining > 0` и декрементит; `RevokeInvite` инкрементит, если код был `pending`; `ConsumeInvite` счётчик не трогает.
- `api-contracts.md`: ответ профиля/настроек включает `invites_remaining`.
