# ADR-0033: Иерархия типов — Pure→Base→Full + производные через утилиты

- Статус: accepted
- Дата: 2026-06-02
- Теги: [architecture] [backend] [frontend] [conventions]
- Вдохновение: `~/coding/WorksControl2/CODE` (libs/common/interfaces) — взята **идея** иерархии, не код (CLAUDE.md #4).

## Контекст

В предыдущей итерации (`~/coding/norms`) типы плодились плоско: для каждого «вида» сущности (создание/чтение/обновление) объявлялся отдельный интерфейс с **повторным перечислением тех же полей**. Поменять тип/опциональность одного поля → искать и править его в нескольких местах; легко рассинхронить. Нужна конвенция, где **каждое свойство объявлено в одном месте**, а «виды» собираются из него.

WorksControl2 решает это иерархией интерфейсов на сущность (`Pure → Base → Full`) + классами через `implements`. Берём принцип, адаптируем под norms2 (Drizzle, строковые id, монорепо **без** workspace).

## Решение

### 1. Иерархия интерфейсов на сущность (каждое свойство — один раз)

Снизу вверх, каждый слой **расширяет** предыдущий и добавляет только новое:

| Слой | Что добавляет | Смысл |
|---|---|---|
| `XxxPure` | содержательные поля сущности (смысл), **без** `id`, FK и системных меток | общий минимум всех «видов» |
| `XxxBase extends XxxPure` | обязательные связи (FK `xxxId: string`) и поля, нужные для создания | форма создания |
| `XxxFull extends Required<XxxBase>` | `id: string` (PK) + системные метки (`createdAt`, `deletedAt?`, …) | полная сущность ≈ строка БД |

### 2. Производные «виды» — ТОЛЬКО через утилиты TS, без переобъявления полей

```ts
type XxxCreate = XxxBase;                          // вход создания
type XxxUpdate = Pick<XxxFull, 'id'> & Partial<XxxBase>;  // частичное обновление
type XxxRead   = Omit<XxxFull, 'passwordHash'>;    // выдача наружу (без секретов)
```

Разрешённый инструментарий: `extends`, `Required`, `Partial`, `Pick`, `Omit`. **Запрещено** повторно перечислять уже объявленные поля в другом интерфейсе вместо наследования/утилиты.

### 3. Классы — через `implements`, не дублируя типы

- DTO (`*.dto.ts`) `implements XxxRead` / `XxxCreate` / `XxxUpdate`. В nest рантайм-валидацию по-прежнему даёт zod-схема ([ADR-0019](./0019-backend-architecture-conventions.md)); интерфейс задаёт **форму типа**, zod — проверку значения (одно описывает другое, не противоречит).
- Доменные сущности/VO используют те же интерфейсы как контракт формы.

### 4. Именование (бест-практис, не венгерское)

- Роль несёт **суффикс** (`Pure/Base/Full/Create/Update/Read`); **без** `I`-префикса и без `IDPK/IDFK`-хвостов (в WC2 были — у нас опускаем).
- PK — `id`, FK — `xxxId`, тип `string` формата `uuidv7___unixmillis` ([ADR-0016](./0016-primary-key-format.md)).
- Файлы kebab-case: `account-base.interface.ts`; barrel `index.ts` на сущность.

### 5. Где живёт — per-project, НЕ шарится

Монорепо **без** workspace (решение Elmir): `angular/` и `nest/` держат типы независимо, общего пакета нет.
- **nest:** `src/interfaces/<entity>/` — полная иерархия (домен — источник истины формы).
- **angular:** `src/app/core/interfaces/<entity>/` (или внутри feature) — **зеркалит подмножество** контракта (`Read/Create/Update`), которое нужно фронту. Дублирование контракта между фронтом и бэком — осознанная цена за независимость сборки/деплоя.

### 6. Стык с Drizzle (домен не знает ORM)

Доменные интерфейсы — источник истины **формы**. Drizzle row-тип (`$inferSelect`) живёт в слое `repositories`; репозиторий **маппит** row → `XxxFull`. На Drizzle-схеме `implements` не вешаем (бизнес-слои не импортируют ORM — [ADR-0030](./0030-stack-revision-drizzle-5layer-npm.md)).

## Зачем (главное)

Свойство объявлено в **одной точке** — в самом нижнем слое, где впервые появляется. Изменение типа/опциональности поля = правка одного интерфейса, изменение распространяется вверх по `extends`/утилитам во все «виды» и реализующие классы. Рассинхрон по определению невозможен.

## Альтернативы

- **Плоские интерфейсы на каждый вид** (как в `~/coding/norms`) — отвергнуто: дублирование, рассинхрон.
- **Общий пакет типов фронт↔бэк** (как libs/common в WC2 через Nx) — отвергнуто: противоречит решению «без workspace, свои package.json».
- **Генерация типов фронта из OpenAPI** — отложено: лишний инструмент в пайплайне на старте; можно вернуться позже.
- **`I`-префикс / венгерские ключи** — отвергнуто в пользу бест-практиса (суффикс роли + `id`/`xxxId`).
