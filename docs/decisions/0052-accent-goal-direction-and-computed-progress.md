# ADR-0052: Цели — семантика `direction`, вычисляемый прогресс и тон forecast

- Статус: принято (2026-06-22, фаза 2, подфаза 2.5·1, реш. Elmir-делегировано) — **частично обратимо** (enum `direction` — аддитивный; формулы — внутренние)
- Контекст: подфаза 2.5 «Цели» (вариант B — полный). Доменная модель (`domain-model.md §4`)
  уже описывает `Goal`/`GoalEntry`/`Milestone`, подцели (C+), вычисляемые поля и rollup. Но
  она исходит из **единственной** семантики значения: `currentValue = Σ GoalEntry.value`. Это
  верно только для целей-**накоплений** («прочитать 50 книг»). Реальные цели бывают трёх родов,
  и от рода зависят формулы `currentValue / percentage / pace / forecast`. До кода это нужно
  зафиксировать — иначе схема и контракт лягут под один частный случай.

Линза 3 шляп (правило нарезки, `impl-phase2-plan.md`):

## 1. Бизнес-аналитик — какие цели ставят люди
Три устойчивых рода цели (наблюдаемо в трекинг-продуктах):
- **Накопление** — «прочитать 12 книг», «пробежать 100 км суммарно», «отложить 100к». Важна
  **сумма** усилий; прогресс монотонно растёт; запись = вклад «+N».
- **Достичь уровня** — «выйти на 15 подтягиваний за подход», «вес 78 кг» (набор массы). Важно
  **последнее измерение показателя**, не сумма; запись = «замер: сейчас X».
- **Снизить** — «вес 75 кг» (с 85), «курить 0 в день», «экранное время ≤ 1ч». То же «достичь
  уровня», но цель **ниже** старта. Психологически — отдельный род (это про вычитание, ядро
  «тренажёра отказа», ADR-0049), и тон forecast тут особенно чувствителен.

Боль, которую решаем: человек хочет **видеть движение к цели честно**, без ручного счёта и без
давящих формулировок. Для нашего таргета (перегруженный человек) цель — якорь смысла, не KPI.

## 2. Системный аналитик — модель, формулы, edge-кейсы

### Решение: `direction` ∈ `accumulate | reach | reduce` (хардкод-enum, не справочник)
Аналогично `HabitKind`/`MicroWinCategory` — закрытое множество, varchar-колонка `direction`.

### Новое поле: `start_value` (numeric, nullable) — базовый замер
- Нужно для `reach`/`reduce`, чтобы считать % (без базы «вес 80→70, сейчас 75» = 107% бессмыслицы).
- Задаётся при создании; **если null** — база = значение **первой `GoalEntry`** (самый ранний замер).
- Иммутабельно после фиксации (фиксированная база, **не счётчик** — гонок нет, `version` не нужен).
- Для `accumulate` — игнорируется (база = 0).

### `currentValue` — вычисляется на чтение (принцип computed-агрегатов, без хранимого счётчика):
- `accumulate`: `Σ GoalEntry.value` (записи — инкременты «+N»; отрицательная = коррекция).
- `reach` / `reduce`: **последняя по `occurred_on`/`created_at`** `GoalEntry.value` (актуальный замер).
  Нет записей → `currentValue = start_value` (или null, если и его нет).

### `percentage` — единая прогресс-доля `f ∈ [0,1]`, **direction-aware**:
- `accumulate`: `f = clamp(current / target, 0, 1)` (start = 0).
- `reach` / `reduce`: `f = clamp((current − start) / (target − start), 0, 1)`.
  Знаменатель `(target − start)` несёт знак → формула едина и для роста (target>start), и для
  снижения (target<start). `reduce` = частный случай `reach` с `target < start`: **математика
  одна**, различаются только (а) тон/копирайт forecast, (б) валидация-подсказки.
- `percentage = round(f · 100)`.

### `forecast` — считается **в пространстве доли `f`** (унифицировано для всех 3 directions):
Пусть `activeDays` = прожитые дни цели минус дни пауз (≥1); `daysLeft` = `deadline − today` (если есть).
- `observedRate = f / activeDays` (доля/день фактически).
- `requiredRate = (1 − f) / daysLeft` (доля/день, чтобы успеть к сроку).
- `forecast` (внутренний enum для цвета UI): `ahead` если `observedRate > requiredRate·(1+ε)`;
  `behind` если `< requiredRate·(1−ε)`; иначе `on_track`. `null` если нет `deadline`, `f≥1`
  (уже достигнута) или `observedRate=0`. (`ε≈0.05`.)
- `projectedCompletionDate = today + (1 − f) / observedRate` дней (если `observedRate > 0`) —
  это и есть «к какой дате дойдёшь при текущем темпе».
- `direction` влияет **только** на то, как получены `current` и `f`; сам forecast — общий код.

### Авто-completed (единственный «modify», без version):
`current` достиг target (`f ≥ 1`) → conditional-update `set status='completed', completed_at=now
WHERE completed_at IS NULL` (идемпотентно, гонки нет). Событие `goal.completed` — эмит отложен до 2.9.

### Rollup (родитель с подцелями) — тоже вычисляемый, без счётчика:
- Если у цели есть **активные** дети → её прогресс = **среднее `percentage` прямых детей**
  (равный вес; объяснимо; обходит проблему смешанных `unit` — у детей разные единицы).
- `currentValue` родителя при rollup **не показываем числом** (единицы детей разнородны) — UI даёт
  `percentage` + «N из M подцелей выполнено». Цель имеет ЛИБО свои entries, ЛИБО детей (если есть
  дети — свои entries у родителя игнорируем для rollup).
- Глубина: `depth < ACCENT_GOAL_MAX_DEPTH` (env, дефолт 2); превышение → `GOAL_MAX_DEPTH_REACHED` 422.

### Edge-кейсы (зафиксировать в коде):
- `reach`/`reduce` без записей и без `start_value` → `currentValue=null`, `percentage=0`, forecast=null.
- `target == start` (reach) → деление на ноль → валидация при создании: `target ≠ start` для reach/reduce.
- `accumulate`: `target > 0`. `reach`: `target` любой ≠ start. `reduce`: `target < start` (подсказка).
- `paused` не принимает entries (`GOAL_PAUSED` 409) и не входит в forecast; `activeDays` исключает паузы.

## 3. Эксперт по помощи людям — тон forecast (никогда не стыдить)
- Внутренний enum `behind` → UI **не** пишет «ты отстаёшь». Копирайт нейтрально-проективный:
  «При нынешнем темпе дойдёшь к ~ДАТЕ» (+опц. мягкое «чтобы успеть к сроку — около X в день»).
- `ahead` → тёплое «идёшь с опережением», без «лучше всех».
- `reduce`-цели (бросить курить и т.п.) — особенно бережно: показываем **сделанный путь** (с 85 до 78),
  не «осталось ещё много». Тон «интерес, не страх» (ui-ux §1) — наша защита от вины.
- `fallbackVersion` (поле цели, текст) — «версия цели на плохой день», операционализирует
  anti-burnout: даже в провале есть минимальная форма движения.

## Решение (итог)
1. **`direction ∈ accumulate | reach | reduce`** (хардкод-enum, varchar-колонка) + **`start_value?`**
   (numeric, иммутабельный базовый замер; для reach/reduce; null → база из первой записи).
2. **Все агрегаты вычисляются на чтение** из `goal_entries`/детей (currentValue/percentage/pace/
   forecast/rollup). `goal_entries` — **append-only INSERT**. Хранимых счётчиков нет → `version`-колонка
   **НЕ нужна** (в отличие от лесенки 2.4). Единственный modify — авто-`completed` через
   conditional-update. `PATCH` полей цели — last-write-wins (низкие ставки).
3. **Прогресс и forecast — в едином пространстве доли `f`**; `direction` влияет только на вычисление
   `current`/`f`. `reduce` = `reach` с `target<start` (общая математика, разный тон).
4. **Rollup родителя = среднее % детей** (равный вес), показываем %+«N из M», без суммы разнородных единиц.
5. **Тон forecast — проективный, не обвиняющий** (см. шляпу 3).

## Обратимость
`direction`/`start_value` — **аддитивные** колонки; добавить 4-й род (например `maintain` — «удерживать
уровень») = новое значение enum + ветка вычисления, без миграции данных. Формулы — внутренние, меняются
без контракта. `reduce`-как-отдельный-enum (а не «reach с target<start») сохранён ради тона/валидации;
схлопнуть его в `reach` позже — тривиально (низкая цена разворота).

## Расширение (2.5·26, 2026-06-24): 4-й род `maintain` — «удерживать уровень»
Реализован как и предсказано выше — **без миграции данных** (переиспользуем существующие колонки):
- **Модель (выбор Elmir — «коридор + adherence»):** `maintain` держит замер в коридоре
  `[startValue, targetValue]` (`startValue`=ниж, `targetValue`=верх; обе обязательны, ниж < верх).
  Покрывает «вес 75–78» и потолки «экран ≤1ч» (ниж=0).
- **`currentValue`** = последний замер (как reach/reduce). **Запись** = замер «сейчас X».
- **Прогресс % = adherence:** доля замеров в коридоре за окно `ACCENT_GOAL_MAINTAIN_WINDOW_DAYS`
  (env, дефолт **30** дней). Нет замеров в окне → `percentage=null` («нет свежих замеров»), без
  фолбэка на всё время (честнее: давно не мерил — не знаешь, держишь ли).
- **forecast/projectedCompletionDate = null** (длится бесконечно; прогноз-к-дате не применим);
  `daysLeft` — если задан срок. **`pace=null`**. **Авто-`completed` НЕТ** (`isGoalReached`→false);
  **вехи не применяются** (`isMilestoneReached`→false).
- **Перф:** батч-метод `maintainAdherenceByAccount` (join `goal_entries`×`goals`, `count filter`)
  считает adherence всех maintain-целей аккаунта одним запросом — без N+1 в списке.
- **Полы «≥X»** маловероятны и закрываются родом `reach` (осознанно не вводим 3-ю границу).
