# claude.md — инструкции для Claude Code

> Этот файл — оперативная инструкция для Claude Code. Читать перед началом любой работы в репозитории.

## TL;DR

1. **Источник истины по продукту** — [`Technical-assignment.md`](./Technical-assignment.md). Не выходить за его рамки без явного запроса.
2. **План работы** — [`todo.md`](./todo.md). Двигаться по фазам, отмечать прогресс там же.
3. **Технические детали** — модульно в [`docs/`](./docs/README.md). Не дублировать, ссылаться.
4. Backend — `./nest/`, frontend — `./angular/`. Не смешивать.
5. Перед коммитом любых изменений — обновить связанные `.md` файлы.

## Как читать репозиторий

Иерархия документов спроектирована так, чтобы агент не тащил всё сразу в контекст:

```
Technical-assignment.md  ← общий обзор, всегда читать первым
        │
        ├─ claude.md     ← как работать (этот файл)
        ├─ todo.md       ← что делать
        │
        └─ docs/         ← подробности по требованию
            ├─ README.md          (индекс — читать вторым, чтобы понять что где)
            ├─ architecture.md    (когда нужна архитектура / границы модулей)
            ├─ domain-model.md    (когда работаешь с сущностями и связями)
            ├─ database.md        (когда пишешь миграции / запросы)
            ├─ api-contracts.md   (когда добавляешь endpoint или DTO)
            ├─ backend.md         (когда работаешь в ./nest/)
            ├─ frontend.md        (когда работаешь в ./angular/)
            ├─ gamification.md    (когда трогаешь очки/серии/достижения)
            ├─ ui-ux.md           (когда верстаешь экран)
            ├─ deployment.md      (когда трогаешь Docker / env)
            └─ getting-started.md (когда что-то не запускается локально)
```

**Правило загрузки:** Claude Code не должен читать всю `docs/` сразу. Читать только то, что нужно для текущей задачи. Использовать `@docs/<file>.md` ссылки.

## Рабочий цикл

При получении задачи:

1. **Уточнить фазу.** Открыть [`todo.md`](./todo.md), найти, к какой фазе относится задача. Если фаза ещё не открыта — сначала закрыть предыдущие.
2. **Прочитать только нужные docs.** По таблице выше выбрать 1–3 документа.
3. **Внести изменения** только в одном слое за раз: либо `nest/`, либо `angular/`, либо `docs/`. Не смешивать в одном коммите backend, frontend и доки.
4. **Обновить документацию,** если изменилось поведение, контракт или структура. Изменение API без обновления `docs/api-contracts.md` — не считается завершённым.
5. **Отметить в `todo.md`,** что пункт выполнен. Если по ходу обнаружились новые подзадачи — дописать их туда же.

## Правила для backend (`./nest/`)

См. подробно [`docs/backend.md`](./docs/backend.md). Кратко:

- Каждая доменная сущность — отдельный модуль (`goals/`, `tasks/`, `anti-habits/`, ...).
- Структура модуля: `*.module.ts`, `*.controller.ts`, `*.service.ts`, `dto/`, `entities/`, `*.spec.ts`.
- Бизнес-логика — в сервисе. Контроллер — только маршрутизация и DTO.
- Миграции пишутся вручную (TypeORM `migration:generate`), не `synchronize: true` нигде кроме явных тестов.
- Все public endpoint-ы помечены через декораторы, валидируют DTO через `class-validator`.
- Любое изменение схемы БД → миграция + обновление [`docs/database.md`](./docs/database.md).
- Любой новый endpoint → запись в [`docs/api-contracts.md`](./docs/api-contracts.md).

## Правила для frontend (`./angular/`)

См. подробно [`docs/frontend.md`](./docs/frontend.md). Кратко:

- Standalone-компоненты, Angular 17+, Signals для локального состояния.
- Структура: `core/` (singletons: auth, http, api), `shared/` (переиспользуемое), `features/<feature>/` (lazy-loaded).
- HTTP-вызовы только через сервисы из `core/api/`. Компоненты не делают `fetch` напрямую.
- Стили — Tailwind. Без inline `style="..."`. Кастомные токены — в `tailwind.config.js`, описаны в [`docs/ui-ux.md`](./docs/ui-ux.md).
- Новый экран → запись в [`docs/ui-ux.md`](./docs/ui-ux.md) с состояниями (loading / empty / error / loaded).

## Где НЕ должно быть кода

- ❌ Не клади бизнес-логику в `app.module.ts` или `main.ts` — только bootstrap.
- ❌ Не клади переменные окружения в код — только через `ConfigService`, описаны в [`docs/deployment.md`](./docs/deployment.md).
- ❌ Не пиши агрегации (проценты, серии, лучший день) на фронте — только запрашивай у бэка.
- ❌ Не используй `synchronize: true` в TypeORM в любом окружении кроме изолированных integration-тестов.
- ❌ Не используй `any` в TypeScript. Если тип неизвестен — `unknown` и сужение.

## Когда обращаться к человеку

Claude Code должен **спрашивать**, а не угадывать, в случаях:

1. Новая сущность не описана в [`docs/domain-model.md`](./docs/domain-model.md).
2. Изменение, ломающее API, без явного указания (versioning policy не определена в этом проекте — bump major не делаем без согласования).
3. Новая внешняя зависимость (npm пакет): сначала проверить, нет ли уже похожей в стеке (см. [`Technical-assignment.md`](./Technical-assignment.md) §3).
4. Что-то противоречит между docs и тикетом — флагнуть несоответствие, не выбирать молча.

Пользователь — Elmir (см. также его другие проекты для понимания стиля: приватный мессенджер, блог-платформа). Любит:

- Серверная агрегация и идемпотентность операций.
- Self-hosted, без облачных зависимостей по умолчанию.

- Магию через декораторы без понимания, что они делают.
- "Удобные" ORM-фичи, которые скрывают SQL (поэтому TypeORM с явными миграциями, а не auto-sync).
- Раздутые UI-фреймворки (поэтому без Angular Material).
