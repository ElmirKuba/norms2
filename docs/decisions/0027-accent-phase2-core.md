# ADR-0027: Ядро Фазы 2 «Акцент» — каркас, лесенка, состояния, состав MVP

- **Статус:** accepted
- **Дата:** 2026-05-30
- **Решает:** Elmir + ToT в [`../archive/phase2-analysis-map.md`](../archive/phase2-analysis-map.md) (развилки R2, R7, R17, R18)
- **Контекст-теги:** [product] [accent] [phase2] [domain]

## Контекст
После переваривания всех источников вдохновения (Ascent + SuperBetter-спек, копилка в phase2-analysis-map §5) фиксируем характер раздела «Акцент». Это раздел внутри Нормисов на общем ЛК фазы 1 (accounts, id `uuidv7___unixmillis`, слоистый Nest, SCSS+MatDialog).

## Решение (по выбору Elmir)

### R18 — каркас сущностей: ГИБРИД
База из Ascent (понятно, измеримо) + усиления SuperBetter:
- **Goal** (= Epic Win): title, **whyItMatters**, unit, targetValue/currentValue, deadline?, **fallbackVersion** (версия цели на плохой день), status, **вехи (Milestones)**.
- **Habit / TaskTemplate→Task** (повторяемые по RRULE) + разовые задачи (= Quest).
- **PowerUp** = «микро-победа» (10 сек–5 мин, выполнима в плохой день, фиксируется как победа) — **техническая форма «1 отжимание = победа»**.
- **AntiHabit** (timer-модель).
- **BadGuy** (препятствие как сущность: trigger/symptoms/counterplay) — берём в MVP (широкий scope).
- **Ally** (союзник поддержки) — через круг доверия/инвайт-дерево.
- **Secret Identity** — развить из существующего `alias` (имя героя/класс/архетип).
- Геймификация: PointEvent (идемпотентный), Achievement, Streak.

### R2 — адаптивная лесенка прогрессии (сердце продукта)
- У каждой привычки/цели есть **«минимальная победа»** (мин. версия). Выполнил минимум → засчитано, **серия сохраняется** (защита от «всё или ничего»).
- Система **предлагает следующий шаг**: легко несколько дней → предлагает больше; тяжело/срыв → держит или откатывает к минимуму.
- **Частичное выполнение** + типы привычек (бинарная / количественная / временная); частичное = микро-победа.
- Цель не «100 отжиманий», а «сделай минимум, остальное бонус».

### R17 — ПОЛНАЯ модель состояний (state-aware)
- Состояния: **survival / recovery / stability / growth / sprint / maintenance** (+ перегружен/откатился).
- **Check-in** (mood/energy/pain/sleep/anxiety/focus) → система выводит состояние.
- **Алгоритм рекомендаций**: по состоянию выдаёт 1–3 микро-действия (survival → 2 power-ups + 1 квест ≤5 мин; growth → больше квестов).
- **Автоснижение нагрузки** при ухудшении; **recovery-режим**; **пауза-режим** (болезнь/отпуск) замораживает серии.
- Срыв = данные, не провал. Без наказания.

### R7 — ШИРОКИЙ MVP (всё из акцента сразу)
**В MVP Фазы 2:** Goal(+why/fallback/вехи) + GoalEntry, Habit/Task (ролловер по RRULE) + разовые, AntiHabit, **PowerUp/микро-победы + адаптивная лесенка**, DailyMetric (сон/энергия/настроение), WeeklyGoal, DailyLesson (урок-якорь), геймификация (XP/серии/достижения), полная модель состояний + check-in, BadGuy, Dashboard-агрегатор.
**Волнами (2.1+):** Workout (тяжёлый модуль), круг доверия/соц (Ally-взаимодействие), AI-помощник, доп-модули (шаблоны/квесты/сезоны/ритуалы/экспорт), полный onboarding.

## Принципы (наши, из обоих источников)
Сначала безопасность→мотивация; микро-действие→амбиция; восстановление→масштаб; доверие→данные. Геймификация поддерживает, не казинит (очки — не валюта, нет gate привилегий). Анти-метрика: НЕ оптимизируем время-в-приложении — цель в реальных изменениях жизни. **Accessibility-first** с самого начала (аудитория — в т.ч. люди в тяжёлом состоянии: депрессия/тревога/ADHD/боль).

## Альтернативы
- Чистый Ascent / чистый SuperBetter — отвергнуты в пользу гибрида.
- Лёгкий state-aware / ядро-MVP — Elmir выбрал полную модель состояний и широкий MVP (богаче, дольше, но цельнее).

## Последствия
- Синтез доки раздела в `docs/sections/accent/`: README (scope+волны), domain-model, gamification (XP/серии/лесенка/состояния), api-contracts, ui-ux.
- На проработку при написании доки (под-ADR или внутри): R1/R10 (атрибуты vs категории целей), R6 (timezone в `accounts` — нужен для ролловера по локальной полуночи), R8 (серии on-demand vs материализация), R11 (формула XP), R20 (Identity из alias), R21 (a11y-требования), R22 (AI-границы).
- Привязка к фазе 1: всё на `accounts`, id-формат, слои, SCSS+MatDialog.
- **Замечание (Claude):** широкий MVP + полная модель состояний = большой объём. При реализации это лучше резать на под-волны 2.0/2.1, но в ДОКЕ опишем всё (принцип Elmir «ничего не упустить, потом шлифовать»).
