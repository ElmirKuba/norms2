# ADR-0019: Конвенции архитектуры бэкенда (C2–C8)

- **Статус:** accepted
- **Дата:** 2026-05-30
- **Решает:** Elmir + рекомендации Claude (пункты C2–C8)
- **Контекст-теги:** [architecture] [backend]

## Контекст
Нужно зафиксировать архитектурные конвенции бэкенда, на которых стоят `architecture.md`, `backend.md`, `api-contracts.md`. Решения согласованы со стеком и предпочтениями (слои, self-hosted, closed shapes, идемпотентность).

## Решение

- **C2 — структура репо:** монорепо. В корне `nest/` (бэк), `angular/` (фронт), `docs/`. У каждого подпроекта свой `package.json` и `.gitignore` (вложенные .gitignore работают). Имена `nest`/`angular` — как в [CLAUDE.md](../../CLAUDE.md).
- **C3 — конфигурация:** `@nestjs/config` + валидация схемы ENV через **zod**, **fail-fast** при старте (нет валидного env → не поднимаемся). В коде env только через `ConfigService`, не `process.env`.
- **C4 — shared kernel:** `nest/src/shared/` — кросс-доменные примитивы: `Id` (VO формата [ADR-0016](./0016-primary-key-format.md)), базовые типы ошибок, `Result`/`Either`, util `generateId()`. Фичи импортят из `shared`, **не** друг из друга.
- **C5 — межмодульное взаимодействие:** модули общаются только через **порты/use-cases application-слоя**. Запрещено лезть в `domain/`/`infrastructure/` чужого модуля. Сохраняет слоистость.
- **C6 — формат ошибок:** единый конверт `{ error: { code: string, message: string, details?: object } }`. Глобальный NestJS exception filter мапит доменные ошибки в HTTP-статус + машинный `code`. Никаких голых строк/стектрейсов наружу.
- **C7 — логирование:** структурный JSON через **pino** (`nestjs-pino`), `request-id` на запрос, без облачных стоков. Тела запросов с плейнтекст-секретами (пароль, ответы) **не логируются** ([ADR-0009](./0009-server-side-hashing.md)).
- **C8 — идемпотентность:** опора на **уникальные ограничения БД** (`lower(login)`, `code`, partial-unique активного бана) — повтор операции не плодит дублей. Мутации спроектированы безопасными к ретраю. Сквозной `Idempotency-Key` middleware — **отложено** до явной нужды (чтобы не тащить стор ключей раньше времени).

## Альтернативы
- joi вместо zod (C3) — zod лучше дружит с TS-типами/closed shapes. RFC 9457 problem+json (C6) — избыточно для фазы 1, свой компактный конверт проще. Winston (C7) — pino быстрее и проще для JSON.

## Последствия
- `architecture.md` — формализует слои, модули, потоки (этот ADR — его основа).
- `backend.md` — детали: zod-схема env, exception filter, pino-конфиг, структура модуля.
- Зависимости: `@nestjs/config`, `zod`, `nestjs-pino`. Проверить отсутствие дублей перед добавлением.
