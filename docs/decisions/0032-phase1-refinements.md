# ADR-0032: Уточнения фазы 1 (валидация, рег-режим, логи, аватарки)

- **Статус:** accepted
- **Дата:** 2026-06-02
- **Решает:** Elmir
- **Контекст-теги:** [phase1] [domain] [api] [frontend] [db]

## Решения

### Длины полей
- **`login`** — 3–32 (без изменений), `^[a-zA-Z0-9_]{3,32}$`, уникальный (по `lower`).
- **`alias`** — **3–32** (было 2–32), Unicode, **не уникальный** (два человека могут иметь одинаковый alias, но не login).
- **`password`** — **3–64** (выбор Elmir).
  - ⚠️ **Зафиксированный риск:** минимум 3 символа для пароля криптографически слаб (мгновенный брутфорс). Принято осознанно (закрытая площадка «для своих», rate-limit на login). При росте/публичности — поднять минимум (отдельный ADR). Макс 64 — в пределах argon2id.
- **`secretAnswer`** — мин 2 (без изменений), нормализация trim+lower+схлоп.

### Рег-режим: свежая проверка при нажатии «Регистрация» (7.1/8)
Проблема: фронт кэширует `feature-flags` с главной; режим мог смениться за дни → юзер заполнит форму и получит ошибку. Решение:
- Новый эндпоинт **`GET /auth/registration-mode`** → `{ mode: 'free' | 'invite' }` (публичный, лёгкий).
- Фронт зовёт его **в момент нажатия «Регистрация»** (не из кэша флагов), и по свежему ответу показывает либо форму (free), либо экран ввода кода (invite).
- `feature-flags` остаётся для общей инициализации; рег-режим — отдельный авторитетный источник на момент действия.

### security_logs — НЕ ведём (2.1)
Таблица `security_logs` **убирается** из фазы 1. Логов безопасности (хеш IP/UA) не пишем. Итого таблиц фазы 1 — **6** (было 7). Rate-limit (`@nestjs/throttler`) остаётся, но работает в памяти/без персистентного следа. Следствие для 152fz: ещё меньше данных — хорошо. Можно вернуть волной отдельным ADR при нужде.

### Профиль — раздельные ответы (7.2)
`GET /accounts/me` — идентичность + inviter + счётчики; список приглашённых — отдельно `GET /invites/mine` (потенциально длинный, пагинируется отдельно). Не объединяем.

### Аватарки — в фазе 1 (10)
- **Кроп на фронте:** пользователь выбирает картинку → обрезает квадрат в модалке (`MatDialog` + компонент аплоада, паттерн Elmir из WorksControl2) → на бэк уходит **уже нарезанное** изображение.
- **Бэк:** принимает готовый файл, валидирует тип (jpeg/png/webp) и размер (лимит в backend.md), сохраняет на диск `content/avatars/<accountId>.<ext>`, путь → `accounts.avatar`.
- **Рендер:** миниатюра аватара в профиле — своём и чужом (`GET /accounts/me`, `GET /accounts/:login` отдают `avatar`).
- Эндпоинты `POST /accounts/me/avatar` (multipart, нарезанное), `DELETE /accounts/me/avatar` — теперь **в MVP** (снимает «задел» из [ADR-0031](./0031-file-storage-uploads.md)).
- Опц. серверная нормализация (re-encode в webp фикс-размера) — рекомендуется для безопасности (срезать EXIF/полезную нагрузку), но кроп — на фронте.

## Последствия (правки доки)
- `database.md`: alias 3–32; убрать таблицу `security_logs` (6 таблиц); `avatar` — не «задел», а активное поле.
- `domain-model.md`: VO `Alias` 3–32, `Password` 3–64 (+пометка риска); убрать упоминание security_logs как сущности.
- `api-contracts.md`: + `GET /auth/registration-mode`; аватар-эндпоинты из «задела» в MVP; убрать security_logs-упоминания.
- `backend.md`: пароль 3–64, лимиты аватара, sharp/обработка опц.; throttler без персист-логов.
- `frontend.md` / `ui-ux.md`: проверка рег-режима по клику; кроп аватара в модалке; рендер миниатюры.
- `152fz.md`: security_logs больше нет — обновить (ещё меньше данных).
