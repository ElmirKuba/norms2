# ADR-0029: Ядро Фазы 3 «НоваСкил» (LMS-раздел)

- **Статус:** accepted
- **Дата:** 2026-05-31
- **Решает:** Elmir (развилки N1–N8 из [`../phase3-analysis-map.md`](../phase3-analysis-map.md))
- **Контекст-теги:** [product] [novaskil] [phase3] [domain] [db]

## Контекст
«НоваСкил» в исходном протоколе (вдохновение, ныне переварено и удалено) — отдельный продукт-LMS со своим ЛК. В Нормисах это **раздел-функционал для аутентифицированных**, наравне с «Акцент». Видение Elmir: Нормисы = площадка «для своих» с подразделами **ЛК / Акцент / НоваСкил / Мессенджер / …**, на **общих аккаунтах и общей БД**.

## Решения

- **N1 — общий ЛК.** НоваСкил использует аккаунты фазы 1 (login+alias, без email), общую БД, общие сессии ([ADR-0018](./0018-refresh-tokens-sessions.md)). Никакого своего ЛК/email/devices. Акцент и НоваСкил параллельны, общая система аккаунтов.
  - _Задел (после MVP, N6):_ «добавить курс в Своё обучение» → отражение в Акценте (курс как Goal/Habit) → переход в курс. Связи UI сейчас нет; в БД таблицы соседствуют, чтобы связать позже.

- **N2 — система ролей (НОВОЕ для платформы).** Вводим:
  - таблица **`roles`** (`id`, `code` uniq: `admin`|`user`, `title`, timestamps);
  - таблица связей **`account_roles`** (`id`, `account_id` FK→accounts, `role_id` FK→roles, уник `(account_id, role_id)`) — один аккаунт ↔ много ролей.
  - Пока 2 роли. **Курсы создаёт/правит `admin`.** Это первая реализация ролей на платформе (в [ADR-0022](./0022-concept-and-philosophy.md) админ-роль была отложена — теперь вводится здесь, платформенно, на будущее годится и для модерации).

- **N3 — контент курсов на диске (как в протоколе).** Структура и тело курсов/глав/уроков — **JSON-файлы на диске** + папка `content/`; в PostgreSQL только то, что про пользователя (роли, прогресс). Это **осознанное отступление** от «всё в TypeORM» ([ADR-0019](./0019-backend-architecture-conventions.md)) ради простоты контент-слоя; зафиксировано здесь.

- **N4 — медиа на диске** (обложки, видео-файлы, картинки уроков, аватары) — загрузка и хранение на диске, как в протоколе. NestJS отдаёт `content/` статикой. (Снимает «отложенность» загрузки файлов из фазы 1 — для раздела НоваСкил решено: диск.)

- **N5 — без UI-связи разделов в MVP.** Акцент и НоваСкил доступны аутентифицированным, **не связаны UI**, но таблицы в одной БД рядом (связать позже дёшево). Нормисы = площадка подразделов, не мессенджер.

- **N6 — связь курс↔Акцент: позже.** В MVP нет. **Задел:** прогресс/enrollment проектируем так, чтобы курс можно было привязать к Goal/Habit Акцента без переделки (общий `account_id`, стабильные id курсов).

- **N7 — двигаемся эффективно.** Основные механизмы и JSON-структуры из протокола берём как продуманные; доку пишем компактно, всю доку обсудим ещё раз потом.

- **N8 — все типы контента сразу:** `md`, `video`, `link`, `html` в уроке с MVP.

## Наследует фундамент (не из протокола — наш стек)
id `uuidv7___unixmillis` ([ADR-0016](./0016-primary-key-format.md), НЕ формат протокола с одним `_`); TypeORM (не Drizzle); монорепо/раздел (не два репо); слои Nest ([ADR-0019](./0019-backend-architecture-conventions.md)); `/api/v1`, конверт ошибок, zod-DTO ([ADR-0020](./0020-api-conventions.md)); Angular standalone+Signals, SCSS, MatDialog ([ADR-0025](./0025-ui-ux-design-language.md), [ADR-0026](./0026-modal-system.md)); тон «интерес не страх», accessibility-first; редактор контента TipTap (headless — ок).

## Альтернативы
- Контент в БД (N3) / медиа отложить (N4) — отвергнуты: Elmir выбрал диск как в протоколе.
- Без ролей (только инвайт-дерево) — недостаточно для «кто правит курсы»; введена таблица ролей.

## Последствия
- Дока раздела `docs/sections/novaskil/`: README, domain-model (БД: roles, account_roles, enrollment/progress + контент-сущности), content-format (JSON на диске), api-contracts, ui-ux.
- **Платформенное:** таблицы `roles` + `account_roles` — общие (не только НоваСкил); упомянуть в `database.md` фазы 1 как платформенное расширение.
- Медиа/контент на диске → в `deployment.md`/бэкапы учесть папку `content/`.
