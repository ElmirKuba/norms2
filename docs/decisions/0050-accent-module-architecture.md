# ADR-0050: Архитектура раздела «Акцент» — мультимодуль (как фаза 1) + правило слоёв

- Статус: принято (2026-06-15, фаза 2, реш. Elmir)
- Контекст: «Акцент» — большой раздел (~12 областей, ~20 таблиц), переплетён
  (геймификация слушает события всех, дашборд агрегирует всех, рекомендатор читает
  многих). Надо решить раскладку модулей и правило применения 5-слойки (ADR-0030).

## Развилка (ToT) — раскладка модулей

| Ветвь | Суть | Минус |
|---|---|---|
| **A. Мультимодуль: область = модуль** (как фаза 1; тонкий зонтик импортит) | консистентно со всем проектом; явные границы/зависимости; сильная инкапсуляция | больше `.module.ts`; нужна дисциплина направлений против циклов |
| B. Компромисс (~6 крупных модулей) | меньше модулей | промежуточный, без явного выигрыша |
| C. Один `AccentModule` + фича-папки | нет межмодульных циклов by construction; меньше файлов | расходится с ф.1; слабая инкапсуляция; большой `providers` |

**Поправка к раннему черновику (важно):** цикл-аргумент против ветви A — НЕ решающий.
В фазе 1 четыре «core»-выноса (`AccessControl` 0037, `BanCore` 0038, `SessionCore`
0043, `NotificationCore`) породил **сквозной Guard** (двунаправлен: нужен всем и сам
нуждается в bans/sessions/account). В «Акценте» такого нет: сквозные места
однонаправленны — геймификация через **события** (не импортит источники), дашборд/
рекомендатор только **читают вниз**. Значит граф мультимодуля ацикличен без «core»-выносов.

## Решение

**Ветвь A — мультимодуль (область = свой NestJS-модуль, как фаза 1).** Выбор ради
**полной консистентности с фазой 1**, явных границ и инкапсуляции. Цикл-риск управляем
(события + однонаправленные зависимости).

- Раскладка: `modules/accent/<area>/<area>.module.ts` (`AccentSettingsModule`,
  `AccentGoalsModule`, `AccentHabitsModule`, …) — каждый со своими слоями
  (`controllers/use-cases/domain-services/adapters/interfaces/dtos`); ORM — в
  `database/repositories/accent/...` + `database/schemas/accent-*.schema.ts`.
- **Тонкий зонтик `AccentModule`** импортит area-модули (чтобы не пухнул `AppModule`);
  можно и регистрировать area-модули прямо в `AppModule` — обе раскладки консистентны.
- `_shared/` раздела — VO (`UserState`/`DomainKey`/`Attribute`) и общие типы, просто
  файлы (без модуля).

**Правило слоёв (применение ADR-0030, «без нюансов»):**
- **Контроллер → use-case — всегда** (use-case = точка приложения/кросс-домена).
- **Кросс-домен и кросс-фаза — только ВНИЗ:** use-case области A зовёт **domain-service**
  области B (или фазы 1), **никогда не use-case B** → граф ацикличен (иначе
  A.use-case ⇄ B.use-case = цикл; ради этого и введён слой domain-services). Направление
  всегда `accent → фаза 1` → межфазовых циклов нет.
- **Свой domain-service / свой порт — по необходимости:** создаём, только если у области
  есть своя логика/данные; иначе use-case композирует чужие domain-services, а
  domain-service использует чужие порты. Тонкий use-case над одним domain-service — норма.
- ORM (Drizzle) — только в `database/`; доменные слои зависят лишь от портов.

## Последствия

- Доступ к фазе 1: area-модуль импортит `AccessControlModule` → `AuthGuard` +
  (реэкспортом) `AccountDomainService`; `InvitesModule` → `InviteTreeDomainService`
  (для `Supporter`); `TransactionRunner` — глобальный. Всё вниз, безопасно.
- Каждая подфаза (2.2 micro-wins, 2.3 habits, …) = новый area-модуль + строка в imports
  зонтика. `2.0.0`: `AccentModule` (зонтик) + `AccentSettingsModule` (первый area-модуль).
- Событийная геймификация (gamification §9) — механизм развязки (EventEmitter2 vs свой
  `DomainEventBus`-порт) выбираем к 2.8; держит граф ацикличным при мультимодуле.
- Дисциплина: следим за направлением импортов (вниз); при появлении двунаправленной
  нужды — выносим «core»-модуль точечно (как ф.1), но по дизайну этого избегаем.
